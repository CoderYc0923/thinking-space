# Docker 容器化部署
Docker 是基于Linux内核虚拟化技术实现的一种容器化技术。

## 核心概念

### Docker引擎
服务器核心运行程序，所有容器的管理者，负责解析镜像，创建容器，调度资源，是部署到额基础环境

### Dockerfile（构建脚本）
配置文件，开发者在文件中定义项目运行规则：运行环境比如node16\jdk1.8、拷贝项目代码、安装依赖、暴露端口、启动命令

### Image（镜像）
通过Dockerfile构建生成的只读静态包，是项目完整运行模板，包含代码、依赖、环境、配置，不可修改，可重复使用、跨服务器迁移。

镜像分层技术：
    1. 原理基于Union File Systems（联合文件系统）的概念实现。
    2. 镜像是由一系列的layers组成，这些层代表文件系统的一系列变化，每一层都包含对文件系统的一个小的改变，比如添加修改删除文件等。分层结构让镜像构建、存储、分发都变得更加高效灵活。
    3. 详解：
        3.1 Base Layer 基础层: 每创建一个新的镜像，会先基于一个基础镜像（比如官方的ubuntu\alpine等）。基础层包含了最基本的操作系统环境，比如Liunx内核和一些基本工具。
        3.2 Intermediate Layers 中间层：在创建镜像的过程中，可以通过Dockerfile添加更多的文件、安装软件包、配置环境变量等。每当执行一个RUN、COPY、ADD等指令，都会创建一个新的层。比如在一个RUN指令中安装了Python，Docker会创建一个新的层来存储Python的安装文件和配置。
        3.3 Final Layer 最终层： 在所有的构建步骤完成后，最终的镜像层包含了运行应用所需的所有文件和配置。这一层通常是最小化的，只包含运行应用所必需的最小文件集。

优化点：多阶段构建（减少最终镜像体积）、精确指定标签（避免latest）、合并RUN指令（减少层数）

Union File System：
    1. Docker使用联合文件系统来管理这些层。最常用的联合文件系统是AUFS（已经被一些新的Docker版本如Docker CE弃用），现在更多的是使用Overlay2或OverlayFS。这些系统允许将多个不同位置的目录叠加，形成一个统一的视图。
        1.1 Overlay2是Docker较新版本中默认使用的文件系统。它通过将多个层合并到一个单一的视图来工作，使得上层目录的内容覆盖下层目录的内容。这种方式非常适合镜像的分层管理，因为你可以很容易地添加或修改文件而不影响其他层。

镜像分层技术优点：
    1. 缓存利用‌：当你再次构建相同的镜像时，如果某些步骤没有改变（例如，安装的软件包版本相同），Docker会重用这些已经存在的层，从而大大加快构建速度。
    2. 共享和重用：由于层的共享，如果你有两个镜像都基于相同的Ubuntu基础镜像，那么它们会共享相同的Ubuntu层，从而减少存储空间的占用。
    3. 可追溯性：每一层的变更都被记录在案，这使得问题排查和镜像审计变得容易。

总结：
    镜像使用分层技术，通过使用联合文件系统有效地管理和优化了存储和性能。每一层的变更都是增量式的，这不仅加快了构建速度，还提高了存储效率，并使得镜像的管理变得更加灵活和可追溯。


### Container（容器）
镜像运行后的动态运行实例，一个Image可以启动多个Container，容器之间相互隔离。

    1. 隔离技术
        1.1 Namespaces 命名空间：利用Linux的命名空间技术来提供独立的系统资源视图，每种类型的资源（如网络、进程、挂载点等）都有自己的命名空间。
            PID命名空间‌：隔离进程树，使得容器内的进程看起来像是运行在独立的系统上
            Network命名空间‌：隔离网络接口，使得容器内的网络与其他容器或宿主机网络隔离
            Mount命名空间‌：隔离挂载点，使得容器内的文件系统与其他容器或宿主机隔离
            UTS命名空间：隔离主机名和域名，允许每个容器有自己的主机名和域名
            IPC命名空间‌：隔离进程间通信（IPC），使得容器内的进程间通信不会干扰其他容器
            User命名空间：隔离用户和组ID，使得容器内的用户和组ID可以与宿主机不同
        1.2 Cgroups 控制组：使用Linux的控制组（Cgroups）来限制、记录和隔离资源（CPU、内存、磁盘I/O等）的使用和分配，避免单个项目占用服务器全部资源，保障多项目稳定共存运行，通过Cgroups，Docker可以确保每个容器获得其所需的资源，同时又不会影响其他容器的运行。
        1.3 Union File Systems‌：容器分层存储，使得容器的文件系统可以共享同一份基础镜像，而只在需要时添加额外的层来存储变化的部分，从而极大地优化了存储空间的使用
        1.4 安全特性：Docker还包括一系列的安全特性来保护容器免受恶意软件的攻击，例如：
            1.4.1 ‌AppArmor‌ 和 ‌SELinux‌ 配置可以限制容器进程可以访问的资源
            1.4.2 Docker Content Trust‌ 用于验证镜像的来源和完整性
            1.4.3 Runtime hooks‌（如seccomp和AppArmor策略）可以进一步限制容器的行为
    
    通过隔离技术，使得应用程序可以在一个安全、资源受限的环境中运行，同时不会影响到宿主机的其他部分或影响到其他容器。这种隔离不仅提高了安全性，也提高了资源管理的效率。

### 网络模型与通信机制
Docker提供五种网络驱动：
    1. bridge（默认，NAT模式）
        1.1 Docker 在宿主机上创建一个虚拟的以太网交换机（虚拟桥接设备），所有的容器都会连接到这个虚拟交换机上
        1.2 每个连接到桥接网络的容器都会被分配一个唯一的 IP 地址，这些地址是在同一个子网内，但与宿主机的 IP 不同
        1.3 容器间的通信通过虚拟交换机进行，外部访问容器需要通过 NAT（网络地址转换）来实现
        1.4 自定义桥接网络
```bash
    # 这条命令创建了一个名为 my_custom_network 的自定义桥接网络
    docker network create --driver bridge my_custom_network
    # 运行容器时指定网络,，在运行容器时，你可以指定它连接到哪个网络。如果不指定，容器将连接到默认的桥接网络
    docker run -d --name my_container --network my_cusom_network nginx
    # 连接外部服务到桥接网络，如果你想要让外部服务（如另一个容器或宿主机上的服务）能够访问到桥接网络中的容器，你可以使用端口映射或者自定义的 DNS 设置来实现，比如访问宿主机的 8080 端口就会重定向到容器的 80 端口
    docker run -d -p 8080:80 --name my_container --network my_custom_network nginx
```
    2. host（共享主机网络栈）
    3. overlay（跨主机网络）
    4. macvlan（物理网络直连）
    5. none（无网络）


## 底层运行原理
Docker并非完整虚拟机，而是轻量级内核级虚拟化，和传统虚拟机原理有本质区别：
传统虚拟机是模拟完整硬件，搭载独立操作系统，占用资源大、启动慢；而Docker容器共享服务器主机的系统内核，仅为每个容器分配独立的文件系统、网络、进程、端口。
依托内核的Namespace隔离机制：每个Docker容器拥有独立的进程空间、网络栈、文件系统，容器内的程序只能访问自身资源，无法影响服务器主机和其他容器，实现项目运行环境的安全隔离。
依托Cgroups资源限制机制：可以精准限制每个项目容器的CPU、内存、磁盘IO占用，避免单个项目占用服务器全部资源，保障多项目稳定共存运行。

## 数据持久化原理
Docker容器默认是临时存储，容器删除后内部数据会丢失。因此生产部署中会通过数据挂载（Volume挂载），将服务器本地目录挂载到容器内部，项目产生的数据库数据、日志、文件会存储在服务器主机目录，而非容器内部，实现数据持久化，保障项目重启、重建容器后数据不丢失。

1. Volume 本质是一个独立存储区域‌：它独立于容器的文件系统，生命周期跟容器分开，容器删了卷还在，数据不会跟着消失 
2. 特点：
    2.1 数据持久化‌：容器重启或删除后，数据还能保留，比如数据库文件、应用配置等
    2.2 数据共享‌：多个容器可以挂载同一个卷，实现数据互通，比如 Web 容器和日志容器共享日志目录 
3. ‌数据实时同步‌：在容器里改卷里的文件，宿主机上能马上看到，反过来也一样，类似双向绑定

三种挂载方式
    1. 管理卷（Volume）：最推荐的方式‌，由 Docker 自动管理，默认存在 /var/lib/docker/volumes/ 目录下 
        1.1 优点‌：移植性高、备份方便、跨平台兼容（Linux/Windows/Mac 都能用）
        1.2 适用场景‌：数据库数据、需要长期保存的应用数据
    2. 绑定卷（Bind Mount）：把宿主机指定目录直接映射到容器里，路径要自己指定 。
        2.1 优点‌：目录好找、方便调试，适合开发时修改代码或配置文件
        2.2 缺点‌：移植性低，不同系统路径结构不一样，Dockerfile 里不能用
    3. 临时卷（tmpfs）‌：数据存在内存里，容器停了数据就没了
        3.1 适用场景‌：高性能临时数据、敏感信息（如密码），不需要持久化的场景 

常用命令
```bash
# 创建卷‌ 不写名字会自动生成随机 ID
docker volume create 卷名

# 查看卷
docker volume ls #列出所有卷
docker volume inspect 卷名 #查看详细信息，包括宿主机上的实际路径

# 挂载到容器
docker run -v 卷名：容器内路径 镜像名 #简单方式
docker run --mount type=volume,source=卷名,target=容器内路径 镜像名 #适合生产环境或需要指定只读等高级选项时

# 删除卷
docker volume rm 卷名 #删除指定卷
docker volume prune -f #强制删除所有未使用的卷
```

### docker compose
用于定义和运行多容器 Docker 应用程序的工具。它使用一个 YAML 文件来配置应用程序的服务。这使得你可以通过一个单独的命令来启动多个容器，而不是手动运行每个容器。

1. 工作原理
    1.1 定义服务：在 docker-compose.yml 文件中，你可以定义一个或多个服务（services）。每个服务可以是一个容器，也可以是一组相关联的容器。例如，你可能有一个前端服务和一个后端服务
    1.2 配置网络‌：你可以配置服务之间的网络连接。默认情况下，docker-compose 创建了一个内部网络，使得在同一 docker-compose.yml 文件中定义的服务可以相互通信。
    1.3 配置卷（Volumes）‌：卷是用于数据持久化的一种方式。你可以在 docker-compose.yml 中配置卷，以便将数据从容器持久化到宿主机或其他卷中。
    1.4 环境变量‌：你可以为服务配置环境变量，这些环境变量可以在容器内被应用程序使用。
    1.5 依赖管理‌：docker-compose 支持定义服务的依赖关系，这样你可以控制服务的启动顺序。例如，你可以确保数据库服务在应用服务之前启动。

2. 优点
    2.1 简化复杂应用的管理‌：通过一个文件管理多个容器和它们的配置
    2.2 易于共享和版本控制‌：docker-compose.yml 文件可以被版本控制，方便团队成员共享和复用配置。
    2.3 快速启动和停止‌：可以一次性启动或停止整个应用程序的所有服务。

3. 配置参数
```bash
‌services‌ #定义一个或多个服务。每个服务可以包含多个配置选项
‌image‌ #指定要使用的 Docker 镜像。例如，nginx:alpine
build‌ #指定 Dockerfile 的路径，用于构建镜像。例如，./dockerfile
ports‌ #映射端口。例如，"80:80" 将容器的 80 端口映射到主机的 80 端口
volumes‌ #挂载卷。例如，./site:/usr/share/nginx/html
environment‌ #设置环境变量。例如，POSTGRES_PASSWORD: example
depends_on‌ #定义服务间的依赖关系。例如，db 服务依赖于 web 服务
networks‌ #定义网络配置。例如，使用默认网络或自定义网络
‌restart‌ #设置重启策略。例如，always 表示总是重启
command‌ #覆盖容器的默认命令。例如，command: python app.py
```

4. 常用命令
```bash
docker compose up #‌启动服务
docker compose up -d #使用 -d 参数可以在后台运行
docker compose down #停止服务
docker compose build #构建服务
docker compose build web db #‌启可以在 docker-compose.yml 中指定要构建的服务
docker compose ps #查看运行中的容器
docker compose logs #查看日志‌
docker compose logs web #‌查看日志‌针对特定服务
```

## 优势

### 环境一致性，零适配成本
镜像固化了所有运行环境，本地开发环境、测试环境、服务器生产环境完全统一，彻底杜绝环境差异导致的部署报错。

### 轻量高效，资源利用率高
共享主机内核，无需搭载独立系统，容器启动仅需秒级，内存、CPU占用远低于虚拟机，一台服务器可部署数十个项目容器，资源利用率最大化。

### 环境隔离，安全稳定
多项目容器相互隔离，A项目的依赖版本、程序故障、漏洞问题不会影响B项目，避免传统部署中多项目环境冲突、互相干扰的问题。

### 快速迁移、扩容、回滚
镜像可打包导出、上传镜像仓库，新服务器只需拉取镜像即可一键部署；需要扩容时可基于同一镜像快速启动多个容器；版本出错时可快速回滚到旧镜像版本。

1. 快速回滚原理
    1.1 镜像管理 通过更新容器使用的镜像标签来指定不同的版本，想要回滚就可以切换到旧版本
    1.2 ‌声明式配置 docker compose 可以更容易地管理和回滚配置变更，允许你定义应用的期望状态，并在需要时轻松地更新这些状态
