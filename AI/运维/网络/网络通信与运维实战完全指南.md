# 第一部分：网络通信核心理论

## 一、网络分层模型：OSI 与 TCP/IP

实际使用的是 **TCP/IP 四层模型**：

| 层级 | TCP/IP 模型 | 常见协议 | 你关注的 |
|------|-------------|----------|----------|
| 4 | 应用层 | HTTP、HTTPS、WebSocket、DNS、SSH | 后端接口、REST API |
| 3 | 传输层 | **TCP**、UDP | 可靠性、连接管理 |
| 2 | 网络层 | IP、ICMP | 路由、ping 排查 |
| 1 | 网络接口层 | 以太网、Wi-Fi | 一般不用管 |

**分层的好处**：每层独立演进。比如 HTTP/3 把传输层从 TCP 换成了 QUIC，应用层代码不用改。

---

## 二、TCP 核心详解

TCP（Transmission Control Protocol）是后端开发最常打交道的传输层协议，**面向连接、可靠、字节流**。

### 2.1 TCP 如何保证可靠性？

| 机制 | 说明 |
|------|------|
| **确认应答（ACK）** | 接收方收到数据后发送确认，发送方没收到就重传 |
| **超时重传** | 发送方启动计时器，超时未收到 ACK 则重传 |
| **序列号** | 每个字节都有编号，接收方可以按序重组、去重 |
| **流量控制** | 滑动窗口机制，接收方告诉发送方自己能收多少 |
| **拥塞控制** | 慢启动、拥塞避免、快速重传、快速恢复四个算法 |

### 2.2 三次握手（建立连接）

```
客户端                       服务端
  |                            |
  |--- SYN=1, seq=x -------->|  ① 客户端：我要连你
  |                            |
  |<-- SYN=1, ACK=1,         |  ② 服务端：收到了，我也要连你
  |    seq=y, ack=x+1 -------|
  |                            |
  |--- ACK=1, seq=x+1,       |  ③ 客户端：好的，连上了
  |    ack=y+1 ------------->|
  |                            |
  |      ESTABLISHED           |      ESTABLISHED
```

**为什么是三次而不是两次？** 防止已失效的连接请求突然传到服务端，导致服务端错误地建立连接。

### 2.3 四次挥手（断开连接）

```
客户端                       服务端
  |                            |
  |--- FIN=1, seq=u -------->|  ① 客户端：我没数据要发了
  |                            |
  |<-- ACK=1, ack=u+1 -------|  ② 服务端：知道了（但可能还有数据要发）
  |                            |
  |<-- FIN=1, seq=v --------|  ③ 服务端：我也没有了
  |                            |
  |--- ACK=1, ack=v+1 ------>|  ④ 客户端：好的，断开
  |                            |
  |   TIME_WAIT (2MSL)        |  ⑤ 客户端等一会儿再释放
```

**TIME_WAIT 为什么重要？**
- 确保最后一个 ACK 能被服务端收到（丢了会重发 FIN）
- 让旧连接的所有报文在网络中消失
- 持续时间：2MSL，Linux 默认约 60 秒

### 2.4 TCP 状态实战价值

| 状态 | 含义 | 常见问题 |
|------|------|----------|
| `ESTABLISHED` | 正常通信中 | ✅ 正常 |
| `TIME_WAIT` | 主动关闭方等待 | ⚠️ 大量出现说明短连接频繁创建关闭 |
| `CLOSE_WAIT` | 被动关闭方未调用 close | 🔴 **代码忘了关连接，会泄露** |
| `SYN_SENT` | 客户端发了 SYN 在等回复 | 🔴 连不上对方，可能防火墙或端口没开 |
| `LISTEN` | 服务端在监听端口 | ✅ 正常 |

### 2.5 TCP 调优参数

在 `/etc/sysctl.conf` 中配置，然后 `sysctl -p` 生效：

```bash
# TIME_WAIT 复用，减少新连接等待
net.ipv4.tcp_tw_reuse = 1

# TIME_WAIT 最大数量，超出的直接关闭
net.ipv4.tcp_max_tw_buckets = 5000

# 建立连接时的 SYN 重试次数
net.ipv4.tcp_syn_retries = 3

# 允许的最大文件句柄数（每个连接占一个）
fs.file-max = 65535

# 端口范围（默认 32768-60999，可调大）
net.ipv4.ip_local_port_range = 1024 65535

# TCP 保活时间（检测死连接，单位秒）
net.ipv4.tcp_keepalive_time = 1200
net.ipv4.tcp_keepalive_intvl = 30
net.ipv4.tcp_keepalive_probes = 3
```

## 三、HTTP 协议演进

| 版本 | 特点 | 现状 |
|------|------|------|
| HTTP/1.0 | 每个请求一个新 TCP 连接 | 几乎淘汰 |
| HTTP/1.1 | **持久连接**（Connection: keep-alive）、管道化 | 目前主流 |
| HTTP/2 | 多路复用、头部压缩（HPACK）、服务器推送 | 越来越多 |
| HTTP/3 | 基于 QUIC（UDP）、0-RTT 握手 | 新兴 |

### HTTP/1.1 的痛点

1. **队头阻塞**：同一个连接上，前一个请求没返回，后面都得排队
2. **并发连接数限制**：浏览器对同一域名一般限制 6-8 个并发连接
3. **头部冗余**：每次请求都带重复的 Cookie 和 User-Agent

### HTTP/2 的改进

- **多路复用**：一个 TCP 连接上同时跑多个请求/响应，互不阻塞
- **头部压缩（HPACK）**：用索引表避免重复传输相同头部
- **服务器推送**：服务端可以主动推资源给客户端
- **运维注意**：HTTP/2 通常需要 HTTPS（TLS），Nginx 配置 `listen 443 ssl http2;`

## 四、HTTPS 与 TLS 握手

HTTP + TLS = HTTPS。TLS 握手大致流程（简化版 ECDHE）：

```
客户端                         服务端
  |                              |
  |--- ClientHello ----------->|  支持的加密套件、随机数1
  |                              |
  |<-- ServerHello ------------|  选定加密套件、随机数2、证书
  |    Certificate              |
  |    ServerKeyExchange        |  ECDHE 公钥参数
  |    ServerHelloDone          |
  |                              |
  |--- ClientKeyExchange ----->|  ECDHE 公钥参数
  |    ChangeCipherSpec         |  后续通信加密
  |    Finished                  |
  |                              |
  |<-- ChangeCipherSpec -------|
  |    Finished                  |
  |                              |
  |    🔒 加密通信               |    🔒 加密通信
```

**核心思想**：对称加密（速度快）和非对称加密（安全但慢）混用——非对称加密只用在握手阶段协商对称密钥，后续数据传输用对称加密。

## 五、WebSocket：全双工通信

HTTP 是请求-响应模式，服务端不能主动推消息。WebSocket 解决这个问题。

### 客户端（浏览器）

```javascript
const ws = new WebSocket('wss://example.com/ws');
ws.onopen = () => ws.send('Hello');
ws.onmessage = (e) => console.log('收到:', e.data);
```

### 服务端（Spring Boot）

```java
@OnMessage
public void onMessage(String message, Session session) {
    session.getBasicRemote().sendText("Echo: " + message);
}
```

### 服务端（Node.js + ws）

```javascript
const WebSocket = require('ws');
const wss = new WebSocket.Server({ port: 8080 });
wss.on('connection', ws => {
  ws.on('message', msg => ws.send(`Echo: ${msg}`));
});
```

**特点**：WebSocket 握手用的是 HTTP Upgrade 机制，建立后切换为全双工二进制协议。常用于**实时消息推送、在线协作、监控大屏**。


## 七、CDN 与反向代理

| 概念 | 说明 | 你的场景 |
|------|------|----------|
| **反向代理** | 客户端→Nginx→后端服务，隐藏真实服务器 | Nginx 负载均衡、SSL 终结 |
| **正向代理** | 客户端→代理→互联网，隐藏客户端身份 | 翻墙、企业内网 |
| **CDN** | 内容分发网络，静态资源就近访问 | 前端静态文件加速 |

## 八、后端开发 Checklist

1. **连接池**：数据库连接、HTTP 客户端都要用连接池，避免频繁 TCP 握手
2. **长连接管理**：HTTP Keep-Alive 的超时时间要合理设置
3. **端口规划**：微服务多了容易冲突，提前规划好端口范围
4. **超时设置**：连接超时、读超时、写超时都要配置，防止资源泄露
5. **健康检查**：每个服务暴露 `/health` 接口给负载均衡器探测
6. **每个接口校验权限**：防止 IDOR 越权漏洞（对比当前用户和请求资源的归属）

## 九、Linux 端命令

### 9.1 核心命令：`ss`（Socket Statistics）

`ss` 是 `netstat` 的现代替代品，速度快、信息全，**几乎预装在所有现代 Linux 发行版**。

#### 常用参数速记

| 参数 | 含义 |
|------|------|
| `-t` | 只看 TCP |
| `-u` | 只看 UDP |
| `-l` | 只看 LISTEN（监听）状态 |
| `-a` | 所有状态（含 ESTABLISHED、TIME_WAIT 等） |
| `-n` | 不解析域名/服务名，显示纯数字（加快速度） |
| `-p` | 显示进程名和 PID（需要 root 权限） |

#### 高频命令

```bash
# 1. 查看所有正在监听的 TCP 端口（最常用）
ss -tlnp

# 2. 查看某个端口被谁占用
ss -tlnp | grep :8080

# 3. 查看所有 TCP 连接（含各状态）
ss -ant

# 4. 查看某进程的所有连接
ss -antp | grep node
ss -antp | grep java

# 5. 统计各状态连接数（排查 TIME_WAIT 过多等问题）
ss -ant | awk '{print $1}' | sort | uniq -c | sort -rn
```

#### 输出解读

```
LISTEN  0  128  0.0.0.0:3306  0.0.0.0:*  users:(("mysqld",pid=1234,fd=23))
│       │   │    │      │         │           │
│       │   │    │      │         │           └─ 进程名、PID、文件描述符
│       │   │    │      │         └─ 远端地址（* 表示接受任意来源）
│       │   │    │      └─ 本地端口
│       │   │    └─ 本地 IP（0.0.0.0 表示监听所有网卡）
│       │   └─ 积压队列上限
│       └─ 已建立的连接数（LISTEN 状态下通常为 0）
└─ 状态
```

### 9.2 命令对照表（当某命令不可用时）

### 9.3 抓包：`tcpdump`

```bash

# -z：只扫描不发送数据
# -v：显示详细输出
```

### 9.7 日志查看

## 十、Windows 端命令

### 10.1 图形化方式：资源监视器（推荐）

最直观的端口排查方式，无需记忆任何命令。

#### 操作步骤

1. **打开**：`Win + R` → 输入 `resmon` → 回车
2. **切换到"网络"标签页**
3. **展开"侦听端口"区域**：
   - 查看所有正在监听的端口
   - 列信息：**映像**（进程名）、**PID**、**地址**、**端口**、**协议**、**防火墙状态**
4. **筛选特定进程**：勾选某进程后，上方的"网络活动"自动只显示该进程的连接
5. **终止进程**：右键 → 结束进程，或记下 PID 去任务管理器关闭

### 10.2 核心命令：`netstat -ano`

```powershell
# 1. 查看某个端口被谁占用（最常用）
netstat -ano | findstr :8080

# 输出示例：
# TCP    0.0.0.0:8080    0.0.0.0:0    LISTENING    12345
# 最后一列 12345 就是 PID
```

### 10.3 根据 PID 查进程名

```powershell
# PowerShell 方式（推荐）
Get-Process -Id 12345

# CMD 方式
tasklist | findstr 12345
```

### 10.4 终止进程

```powershell
# PowerShell
Stop-Process -Id 12345 -Force

# CMD
taskkill /PID 12345 /F
```

### 10.5 一行命令直接干掉占用某端口的进程

```powershell
# PowerShell 5.1+（Get-NetTCPConnection 比 netstat 更强）
Get-NetTCPConnection -LocalPort 8080 -ErrorAction SilentlyContinue | ForEach-Object {
    Stop-Process -Id $_.OwningProcess -Force
}
```

### 10.6 Windows vs Linux 命令对照

| 你想知道什么 | Linux 命令 | Windows 命令 |
|-------------|-----------|-------------|
| 端口被谁占用 | `ss -tlnp \| grep :8080` | `netstat -ano \| findstr :8080` |
| 所有监听端口 | `ss -tlnp` | `netstat -ano \| findstr LISTENING` |
| 进程打开了哪些连接 | `ss -antp \| grep 1234` | `netstat -ano \| findstr 1234` |
| 查进程名（按 PID） | `ps -p 1234` | `tasklist \| findstr 1234` |
| 杀进程 | `kill 1234` | `taskkill /PID 1234 /F` |
| 路由追踪 | `traceroute` / `mtr` | `tracert` |
| DNS 查询 | `dig` / `nslookup` | `nslookup` |
| 连通性测试 | `nc -zv` / `telnet` | `Test-NetConnection` / `telnet` |

# 第三部分：实战场景速查

## 场景 1：启动应用时提示"端口已被占用"

```bash
# Linux
ss -tlnp | grep <端口号>
# 找到 PID 后：
kill -9 <PID>

# Windows（PowerShell）
netstat -ano | findstr :<端口号>
taskkill /PID <PID> /F
```

## 场景 2：怀疑数据库连不上

```bash
# 测试端口是否通
nc -zv <数据库IP> 3306

# 或者用 telnet
telnet <数据库IP> 3306

# 成功则显示 Connected，失败则超时或 Connection refused
```

## 场景 3：线上服务器 TIME_WAIT 过多

```bash
# 先看数量
ss -ant | grep TIME_WAIT | wc -l

# 如果成百上千，调内核参数
sysctl -w net.ipv4.tcp_tw_reuse=1
sysctl -w net.ipv4.tcp_max_tw_buckets=5000

# 持久化到配置文件
echo "net.ipv4.tcp_tw_reuse = 1" >> /etc/sysctl.conf
echo "net.ipv4.tcp_max_tw_buckets = 5000" >> /etc/sysctl.conf
sysctl -p
```

## 场景 4：服务 CLOSE_WAIT 堆积

这是 **代码 BUG**，不是系统问题：应用收到了对端的 FIN 但忘记调用 `close()`。

排查：
```bash
# 看哪个进程的 CLOSE_WAIT 多
ss -antp | grep CLOSE_WAIT
```
然后去对应的 Java/Node.js 代码里检查连接关闭逻辑（数据库连接池、HTTP 客户端等）。

## 场景 6：Nginx 日志排查

```bash
# 实时查看访问日志
tail -f /var/log/nginx/access.log

# 查看错误日志
tail -f /var/log/nginx/error.log

# 统计访问量最高的 IP
cat access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head -10

# 统计状态码分布
cat access.log | awk '{print $9}' | sort | uniq -c | sort -rn
```
