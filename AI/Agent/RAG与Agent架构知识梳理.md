# RAG 与 Agent 架构知识梳理

> 整理自多轮对话讨论，涵盖 RAG、知识库、向量数据库、Agent 架构等核心概念及其关系。

---

## 一、RAG（检索增强生成）是什么？

### 1.1 一句话定义

**让 AI 在回答前，先去"查资料"，再结合资料生成答案。**

传统大语言模型像闭卷考试的学霸——知识截止于训练数据，不知道训练后的新信息，也无法访问私有数据。

### 1.2 工作流程

```
用户提问 → 检索相关文档/资料 → 将检索结果 + 问题一起给 AI → AI 生成答案
```

**举例**：你问"女主的黑暗元素能力有什么限制？"——RAG 先到你的设定文档里搜索相关段落，找到后再让 AI 基于真实设定回答，而不是让 AI 自己编。

## 二、RAG 由什么组成？

RAG 不是单一技术，而是一条**流水线（Pipeline）**，共五个阶段：

### 2.1 完整组成图

```
┌─────────────────────────────────────────────────┐
│                   RAG 流水线                      │
├───────────┬──────────┬──────────┬──────┬────────┤
│  ①文档    │  ②切割   │  ③向量化  │ ④检  │ ⑤生成  │
│  加载     │  分块    │  建索引   │  索   │  增强  │
├───────────┼──────────┼──────────┼──────┼────────┤
│ PDF/MD/   │ 把长文   │ Embedding│ 用户  │ 检索结 │
│ 网页/API  │ 切成小   │ 模型把每 │ 提问→ │ 果+问题│
│ → 原始    │ 段落     │ 块转成向 │ 相似  │ →拼成  │
│   文本    │ (chunk)  │ 量存入   │ 度搜  │ prompt │
│           │          │ 向量数   │ 索→   │ →LLM   │
│           │          │ 据库     │ Top-K │ 生成   │
└───────────┴──────────┴──────────┴──────┴────────┘
```

### 2.2 各阶段详解

#### 阶段一：文档加载
把 PDF、Markdown、网页、API 返回数据等原始文本加载进来。

#### 阶段二：切割分块（Chunking）
把长文档切成小段落（chunk），比如把 900 章的剧情大纲切成每 200 字一段。

#### 阶段三：向量化建索引
用**嵌入模型（Embedding Model）** 把每个文本块转成一串数字（向量），存入**向量数据库**。语义越近的文本，向量距离越近。

常用嵌入模型：`text-embedding-3`、`bge-large` 等。

#### 阶段四：检索
用户提问也被转成同样的向量格式，在向量数据库里做相似度搜索，取 Top-K 最相关的文档块。

- 检索策略：不只是纯向量相似度，还可以混合关键词搜索、多轮检索
- 重排序模型（Reranker）：对检索到的 Top-K 做一次精排，去掉噪音

#### 阶段五：生成增强
把检索到的文档块 + 用户问题 + 系统提示拼成完整 prompt，交给 LLM 生成最终答案。

## 三、知识库 vs 向量数据库

### 3.1 核心区别

| | 知识库 | 向量数据库 |
|---|---|---|
| **是什么** | 原始文档的集合 | 向量化后的语义索引存储 |
| **存什么** | PDF、Markdown、网页原文、表格 | 一串数字（向量）+ 对应原文引用 |
| **人能读吗** | ✅ 能 | ❌ 数字串，人看不懂 |
| **机器能搜吗** | 只能关键词搜索 | ✅ 语义搜索 |
| **类比** | 图书馆的书架 | 图书馆的索引卡片柜 |

### 3.2 它们的关系

```
知识库（原始文档）→ 切割分块 → 嵌入模型 → 向量数据库
     ↑                                          ↓
  （文档源头）                          （语义索引层，
                                         支撑快速相似度搜索）
```

> **"向量知识库"** 不是一个标准术语，通常指的就是 **向量数据库 + 它索引的知识库** 这两个东西的合称。

### 3.3 完整数据流

```
        知识库（原始文档）
        ├── 世界设定v3.md
        ├── 人物画像.md
        ├── 剧情大纲.md
        └── ...
              │
              ▼ 文档加载 + 切割分块
        ┌─────────────────────────┐
        │  文本块(chunks)          │
        │  "女主第2年进入异度..."  │
        │  "灵宠仅为媒介无能力..." │
        │  "赛季=100天..."         │
        └─────────────────────────┘
              │
              ▼ 嵌入模型（Embedding Model）
        ┌─────────────────────────┐
        │  向量数据库              │
        │  [0.23, -0.87, ...] → chunk_001
        │  [0.65, 0.12, ...]  → chunk_002
        │  [0.98, -0.33, ...] → chunk_003
        └─────────────────────────┘
              │ 用户问题也转成向量 → 相似度搜索
              ▼
        ┌─────────────────────────┐
        │  检索结果（Top-K 原文）   │
        │  最相关的3段设定文本     │
        └─────────────────────────┘
              │ 拼入 prompt
              ▼
        ┌─────────────────────────┐
        │  LLM 生成回答            │
        │  基于你的真实设定回答    │
        └─────────────────────────┘
```

## 四、Agent 架构中 RAG 的位置

### 4.1 Agent 四大模块（Lilian Weng 经典总结）

Agent = **规划 + 记忆 + 工具 + 行动** 四大模块。

| 模块 | 职责 |
|---|---|
| **规划（Planning）** | 推理、决策、拆解子目标 |
| **记忆（Memory）** | 短期记忆（对话上下文）+ 长期记忆（RAG 知识库） |
| **工具（Tools）** | 调用外部 API、数据库、代码执行器 |
| **行动（Action）** | 实际执行操作，返回观察结果 |

> 注意：Coze/Dify 等平台常把四大模块归并为"大脑 + 规划 + 记忆 + 工具"，把行动合并到工具里。本质覆盖的是同一套东西，只是归并方式不同。

### 4.2 Agent 三大层（偏软件架构）

```
┌─────────────────────────────────────┐
│           编排层（大脑）              │  ← 规划、推理、决策
├─────────────────────────────────────┤
│           记忆层                    │  ← 短期记忆 + 长期记忆（RAG 核心所在）
├─────────────────────────────────────┤
│           工具层（手脚）             │  ← 调用外部工具、API、数据库
└─────────────────────────────────────┘
```

### 4.3 Agent 三大层（偏控制论/机器人视角）

```
感知层 → 决策层 → 执行层
  ↑        ↑        ↑
接收输入  大脑推理  实际调用工具
检索文档  规划决策  生成回答
```

| 层 | 职责 |
|---|---|
| **感知层** | 接收输入（用户问题、传感器数据、**检索到的文档**） |
| **决策层** | 大脑推理、规划、决定下一步做什么 |
| **执行层** | 实际调用工具、生成回答、操作环境 |

### 4.4 两套三层模型的关系

两套三层不是互斥的，而是**正交的**：

```
              感知层         决策层         执行层
编排层    ← 输入路由到哪  ← LLM怎么调度  ← 结果怎么分发
记忆层    ← 检索相关记忆  ← 筛选Top-K    ← 写入新记忆
工具层    ← 工具返回结果  ← 判断调哪个工具 ← 实际执行调用
```

### 4.5 RAG 在不同框架中的位置

| 用哪套框架 | RAG 的位置 |
|---|---|
| **编排/记忆/工具三层** | 横跨记忆层和工具层，核心在记忆层的检索能力 |
| **感知/决策/执行三层** | 主要在**感知层**：从知识库检索相关内容，作为输入的一部分 |

> **总结**：RAG 不管用哪套框架，核心位置都是一样的——**连接"外部知识"和"模型推理"之间的那道桥梁**。有时被封装为记忆层的核心能力，有时被封装为工具层的一个 Tool。

## 五、RAG 中容易被忽略的组件

| 组件 | 作用 | 备注 |
|---|---|---|
| **嵌入模型** | 把文本变成向量，语义越近向量越近 | text-embedding-3、bge-large 等 |
| **检索策略** | 混合搜索：向量相似度 + 关键词 + 多轮检索 | 确保召回率和精确率 |
| **重排序模型** | 对检索到的 Top-K 再做一次精排，去掉噪音 | 确保"黑暗元素"比"黑暗料理"更优先 |
| **上下文窗口管理** | 检索结果太长时，截断、压缩、分层处理 | 900 章大纲不能一次性全塞进 prompt |

## 六、实际应用举例：小说写作工作流

### 6.1 你的写作场景对应

| RAG 组件 | 你的写作工作流对应 |
|---|---|
| **知识库** | 大纲、细纲、人物画像、世界设定、语言风格等所有创作文档 |
| **嵌入模型** | 把设定文档转成向量索引 |
| **向量数据库** | 存储所有设定的语义索引，支持快速检索 |
| **检索** | 每次写作或审稿时，自动检索相关设定（如人物不 OOC、剧情不走偏） |
| **生成** | AI 基于检索到的设定生成正文或审查意见 |

### 6.2 你当前的"手动 RAG"

你每次提问时，AI 调用的"已有记忆"（11 条设定记忆），本质上就是一个微型 RAG：
- 知识库 = 你的设定文档
- 检索 = AI 查记忆
- 生成 = 基于记忆内容回答

---

## 七、一句话总结

> **RAG = 知识库（原始文档）+ 嵌入模型（文本→向量）+ 向量数据库（语义索引）+ 检索引擎（相似度搜索）+ LLM（生成回答）。**
>
> 在 Agent 架构中，RAG 横跨记忆层和工具层，本质是连接外部知识与模型推理的桥梁，无论用哪种架构框架切分，核心位置不变。
