# Agent 开发中的流程编排模式：链式、DAG 与图

> 在 Agent 开发中，链式（Chain）、DAG（有向无环图）和图（通常指带循环的状态图）是三种核心的流程编排模式。它们分别对应不同的控制粒度、灵活性和复杂度，理解其原理与差异是设计可靠 Agent 系统的关键。

---

## 一、链式（Chain）

### 原理

链式是将多个处理步骤按线性顺序串联，前一步的输出直接成为后一步的输入，形成一个单向的、无分支的执行流。

```
Step A → Step B → Step C → ...
```

所有步骤依次执行，没有并行或条件跳转（高级封装可能在内部含路由，但结构上仍是单一的链）。

### 优势

- **简单直观**：逻辑清晰，易于编写、测试和调试。
- **确定性**：执行路径唯一，结果可预期，排错方便。
- **低开销**：无需复杂的调度或状态管理。

### 劣势

- **刚性**：无法利用并行加速，也难以应对多分支、条件动态决策等场景。
- **复用性弱**：步骤与整条链高度耦合，拆解和重组不便。

### 应用场景

- 简单的 RAG 流水线：检索文档 → 拼接提示词 → 调用 LLM 生成答案。
- 数据处理 ETL：读取 → 清洗 → 标准化 → 存储。
- 无需决策分支的固定流程，如文档摘要生成。

### 典型框架

LangChain 的 `LLMChain`、`SequentialChain`；早期简单 Agent 的线性编排。

---

## 二、DAG（有向无环图）

### 原理

DAG 由节点（任务）和有向边（依赖关系）构成，图中不存在环路。每个节点可以有多个前驱和多个后继，执行引擎会根据依赖关系自动调度：

- 无依赖的节点可**并行执行**
- 分支节点可根据**条件路由**到不同下游
- 汇聚节点等待所有前置完成后再触发

常见结构：`条件分支 → 并行处理 → 结果合并`

### 优势

- **并行与效率**：互不依赖的节点可以同时执行，显著降低端到端耗时。
- **灵活分支**：支持条件边（如按意图分流不同处理逻辑），比链式更适应多变业务。
- **可观测性强**：DAG 的拓扑结构清晰，可追踪每一步的输入输出和延迟。
- **确定性**：无环保证了流程终会结束，不会陷入死循环。

### 劣势

- **无法循环**：不能应对需要反复尝试或动态多轮的自主推理（如 Agent 持续思考-行动）。
- **设计复杂度提升**：需要显式定义依赖和合并逻辑，节点间的数据传递需规范。

### 应用场景

- 复杂多步骤的 RAG：查询重写 → 多路检索（向量、关键词、知识图谱）→ 结果融合 → 生成。
- 多工具并行调用：同一问题分发到搜索、计算器、数据库，聚合结果后回答。
- 分阶段内容生成：大纲生成 → 各章节并行写作 → 汇总润色。
- 明确无循环的任何确定性工作流。

### 典型框架

- **Haystack 2.x** 的 `Pipeline` 本质上就是 DAG
- **LangGraph** 可构建无环图（即 DAG）
- **Airflow**、**Prefect** 等任务调度系统也以 DAG 为核心（但偏向数据工程）

> 在 Agent 语境下，DAG 适合"一次性"决策和执行的场景，无需 Agent 反复自我纠错。

---

## 三、图（带循环的状态图）

### 原理

此处的"图"特指**允许环路**的有向图，通常实现为**状态图**。它围绕一个共享状态对象，节点（处理步骤）读取/更新状态，边代表控制流，其中：

- **条件边**可以根据状态动态决定下一节点
- 甚至可以指回上游节点，**形成循环**

这种循环能力支撑了经典的 **ReAct 循环**：

```
Agent 推理 → 行动（调用工具）→ 观察结果 → 再推理 → ...
```

直到满足终止条件。图中常包含特殊节点（如 END）和持久化机制，支持暂停、恢复和人机协同。

### 优势

- **自主性与灵活性**：可实现动态规划、多步推理和错误修正，Agent 能根据环境反馈调整行为。
- **循环与迭代**：天然适合需要未知步骤数的探索性任务（如长对话、自主研究）。
- **状态持久化**：状态图可检查点保存，支持暂停/恢复、回放、分支执行（类似"平行宇宙"时间线），便于调试和人类审批。
- **可建模极其复杂的逻辑**：能实现分层 Agent、多 Agent 协作等高级模式。

### 劣势

- **复杂度高**：设计、调试和监控难度远高于链式或 DAG，易出现无限循环、状态膨胀等问题。
- **非确定性**：执行路径依赖动态状态，输出重现需要严格的状态管理。
- **资源消耗**：循环和持久化需要更多计算和存储资源。

### 应用场景

- **自主 Agent**：如 AutoGPT、LangGraph Agent，能持续使用工具、反思结果、修正计划。
- **复杂交互式对话**：需要多轮澄清、分支引导、外部系统调用的客服机器人。
- **多 Agent 协作**：多个 Agent 轮流执行，通过共享状态"协商"，形成循环依赖的协作网络。
- **需要人在回路的流程**：循环中暂停等待人工审核，审核后继续执行。
- **长期运行的探索任务**：代码生成中写-跑-修循环，或数据科学中假设验证循环。

### 典型框架

- **LangGraph** — 当前最主流的实现，其 `StateGraph` 直接支持循环、条件边和状态持久化。它也是 LangChain 生态中从"链"向"图"演化的核心工具。
- OpenAI 的 **Swarm**（轻量级）
- **CrewAI** — 可构建带循环的 Agent 协作图

## 四、三者核心区别

| 特性 | 链式 (Chain) | DAG | 图 (Graph，带循环) |
|------|:-----------:|:---:|:-----------------:|
| **拓扑结构** | 线性，单链表 | 有向无环，可分支合并 | 有向有环，任意连接 |
| **循环** | ❌ 无 | ❌ 无 | ✅ 支持 |
| **并行** | ❌ 不支持 | ✅ 支持无依赖节点并行 | ✅ 支持（但循环内通常串行） |
| **动态决策** | 弱（往往需预先编码） | 中等（条件边可路由） | 强（依赖状态的条件边 + 循环） |
| **状态管理** | 仅传递数据 | 节点间传递数据 | 全局共享状态，可持久化 |
| **可观测性** | 简单 | 较好，可追踪 DAG 执行 | 复杂，需专门工具（如 LangGraph Studio） |
| **设计思维** | "做这个，然后做那个" | "这些任务按依赖运行，可并行" | "在这个状态下，下一步做什么？可能循环" |
| **典型 Agent 模式** | 固定流水线 | 多步确定工作流 | ReAct、Plan-Execute、多 Agent 循环 |

### 包含关系

```
链式 ⊂ DAG ⊂ 图
```

- **链式**是只有一条路径的 DAG
- **DAG** 是无环的图
- **带循环的图**是最通用的模型，LangGraph 等框架也支持退化为 DAG 或链式使用

---

## 五、如何选择？

| 场景 | 推荐模式 |
|------|---------|
| 任务极其简单，步骤固定，不需要分支或并行，追求最快的开发速度和最易的维护 | **链式** |
| 需要明确的多个步骤，且这些步骤间有并行机会或条件分流（如根据意图执行不同工具），但流程在一次性执行后就能得到最终结果，无需 Agent 反复自我修正 | **DAG** |
| 任务需要 Agent 自主决策、使用工具并从反馈中学习，步骤数不固定，可能涉及多轮交互、审批或复杂的多 Agent 协作 | **图（带循环）** |

---

## 六、总结

在现代 Agent 开发中，趋势是使用**支持循环的图框架**（如 LangGraph）统一编排，因为它可以灵活地定义链式、DAG 或复杂循环，并自带状态持久化。对于简单部分，仍可内部封装为链式或 DAG 子图，兼顾开发效率与灵活性。

```
灵活性：链式 < DAG < 图
复杂度：链式 < DAG < 图
自主性：链式 < DAG < 图
```

**核心建议**：从需求出发选择最简模式，避免过度设计。能用链式解决的，不必上 DAG；能用 DAG 解决的，不必引入循环。但当业务确实需要 Agent 自主推理和迭代时，毫不犹豫地使用带循环的图。
