# 大模型开发：Workflow vs Agentic Workflow vs Agent 完整指南

## 一、概述

三者的核心区别在于 **"谁决定下一步做什么"**：

- **Workflow**：代码说了算，LLM 是"计算器"
- **Agentic Workflow**：骨架代码定，路口模型选，LLM 是"交警"
- **Agent**：模型说了算，LLM 是"司机"

```
┌─────────────────────────────────────────────────────────────────┐
│                    自主性递增 + 确定性递减                        │
│                                                                 │
│  Workflow              Agentic Workflow             Agent       │
│  (确定性流程)          (流程中嵌入智能决策)         (全自主推理)   │
│                                                                 │
│  人决定下一步          关键节点 LLM 决策            LLM 决定每一步 │
│  LLM 是"计算器"        LLM 是"路口交警"            LLM 是"司机"   │
│                                                                 │
│  ───────────────── 控制权逐渐向模型转移 ─────────────────       │
└─────────────────────────────────────────────────────────────────┘
```

**决策口诀**：
> 能用 Workflow 的，不用 Agentic Workflow。
> 能用 Agentic Workflow 的，不上完整 Agent。
> 只在路径确实无法穷举时，才放 Agent 自主探索。

---

## 二、Workflow（纯工作流）

### 定义

**代码说了算，不是模型说了算。** 整个执行图在编码阶段就固定了，LLM 只是其中一个"文本处理工具"，不是决策者。

### 类比

工厂流水线——工件在第 3 步被打螺丝，不是因为 AI 判断需要打，而是因为工艺图纸就是这么写的。

### 核心特征

- 控制流：预定义图（DAG/Sequential/Parallel）
- LLM 角色：计算单元（处理文本、提取实体、生成摘要等）
- 决策权：100% 在开发者手里
- 确定性：极高，路径恒定
- Token 消耗：低
- 调试难度：低

### 经典例子

- **传统 RAG 管道**：检索文档 → 拼 Prompt → 调用 LLM → 返回结果
- **数据清洗 ETL**：读 CSV → 调 LLM 做实体抽取 → 写入数据库
- **定时报告生成**：每天 8:00 从数据库取数据 → 生成报告 → 发送邮件

### 在 pi-agent-core 中的映射

`ControlFlow` 层直接编排多个 `WorkflowUnit`，按 Sequential/Parallel/DAG 执行，中间没有 Steering 干预点。

## 三、Agentic Workflow（智能工作流）

### 定义

**流程骨架固定，但关键路口让 LLM 做选择。** 在当前业界最主流、最务实的范式，把 Workflow 的可靠性和 Agent 的灵活性结合起来。

### 类比

高速公路 + 智能导航——路是修好的（Workflow 骨架），但每个分岔口导航会根据实时路况（上下文）选择走哪条匝道（LLM 路由）。

### 核心特征

- 控制流：骨架预定义，分支点由 LLM 动态决策
- LLM 角色：路由器/决策者（选择走哪条分支）
- 决策权：开发者定骨架，模型做选择题
- 确定性：中高，骨架不变
- Token 消耗：中
- 调试难度：中

### 三种经典形态

#### 形态一：Router Workflow

入口处 LLM 做一次意图分类，把请求分发给不同的下游子流程。

```
用户输入 → LLM 路由节点(分类) → 客服子流程 / 查询子流程 / 售后子流程
```

**例子**：用户问"我的订单怎么还没到？"
1. LLM 路由节点判断：这是"物流查询"类 → 转给物流查询子流程
2. 物流查询子流程是纯 Workflow：调物流 API → 格式化 → 输出

#### 形态二：Plan-Execute

Agent 先生成一个步骤列表（Plan），然后按图索骥依次执行，每步结果反馈回 Planner。

```
用户目标 → LLM Planner 生成步骤列表 → Sequential 逐步执行 → 结果汇总
```

**例子**："帮我写一份 Q3 销售分析报告"
1. Planner 拆解：①查 Q3 销售数据 → ②计算同比环比 → ③生成图表 → ④撰写报告
2. 外层 Sequential 按序执行每一步

#### 形态三：Human-in-the-Loop

流程运行到关键节点时暂停，等待人工审批后再继续。

```
正常流程 → 关键节点(金额 > 1000) → 暂停 → 人工审批 → 继续/驳回
```

**例子**：退款金额 > 500 元时，暂停等待客服主管审批。

### 为什么 Agentic Workflow 是当前最佳实践

- ✅ 确定性骨架保证业务逻辑不跑偏
- ✅ LLM 只在明确的决策点介入，消耗可控
- ✅ 可测试、可审计（因为骨架固定）
- ✅ 复杂决策交给模型，避免 if-else 地狱

### 在 pi-agent-core 中的映射

- Router Workflow → `ControlFlow.Router` + `Steering` 决定路由
- Plan-Execute → `WorkflowUnit` 内 Agent 产出计划，外层 `Sequential` 执行
- Human-in-the-Loop → `Steering/FollowUp` 机制，外部事件驱动推进

## 四、Agent（全自主智能体）

### 定义

**给目标 + 工具箱，让模型自己探索路径。** 没有预设的流程图，Agent 在"思考→行动→观察→再思考"的循环中自主决定一切。

### 类比

给一个实习生一个目标和一把钥匙，他自己在楼里找路——他可能走错、可能走远路、也可能找到你没料到的捷径。

### 核心特征

- 控制流：完全动态，无预设图
- LLM 角色：调度中心（指挥一切工具和步骤）
- 决策权：几乎 100% 在模型手里
- 确定性：低，每次可能不同
- Token 消耗：高（多轮 ReAct 循环）
- 调试难度：高（需 trace 推理链）

### 经典例子

- **旅行规划 Agent**："帮我订下周去北京的出差行程"→ 查日历、搜航班、比价、订酒店，中间发现航班售罄再换方案
- **客服 Agent**："我的订单怎么还没到"→ 自己查订单状态、查物流 API、决定退款还是催件
- **代码助手 Agent**："给我写一个用户登录模块"→ 自己读项目结构、写代码、写测试、运行测试、修复报错

### 在 pi-agent-core 中的映射

`WorkflowUnit` 内部的 `AgentLoop`（ReAct 引擎），无外层 ControlFlow 约束。

## 五、三者完整对比表

| 维度 | Workflow | Agentic Workflow | Agent |
|------|----------|------------------|-------|
| **控制流图** | 完全预定义 DAG | 骨架固定，分支动态 | 完全动态，无预设图 |
| **谁决定下一步** | 代码 | 代码 + LLM 选分支 | LLM |
| **LLM 角色** | 计算单元（处理文本） | 路由器/决策者 | 调度中心（指挥一切） |
| **工具调用** | 固定位置、固定顺序 | 在决策点后触发预设工具集 | 自主选择工具和顺序 |
| **确定性** | 极高，路径恒定 | 中高，骨架不变 | 低，每次可能不同 |
| **Token 消耗** | 低 | 中 | 高 |
| **调试难度** | 低 | 中 | 高 |
| **灵活性** | 低 | 中 | 高 |
| **工程成熟度** | 极成熟 | 当前主流 | 快速发展中 |
| **典型实现** | LCEL Chain | Router/Plan-Execute/HITL | ReAct Agent |
| **pi-agent-core** | ControlFlow 层 | ControlFlow + Steering | AgentLoop |

## 六、混合架构示例

真实系统中，三者的关系不是"三选一"，而是**在同一个应用的不同层级上混合使用**。

```
┌──────────────────────────────────────────────────────────┐
│                      应用层                              │
│  ┌────────────────────────────────────────────────────┐  │
│  │              Agentic Workflow (顶层路由)            │  │
│  │                                                    │  │
│  │   用户输入 ──→ LLM Router ──→ 分类意图              │  │
│  │                  │                                 │  │
│  │     ┌────────────┼────────────┐                   │  │
│  │     ▼            ▼            ▼                   │  │
│  │  客服子流程   查询子流程   售后子流程                │  │
│  │  (Workflow)  (Workflow)  (Agent)                  │  │
│  │                          ┌──────────────────┐     │  │
│  │                          │  退款金额 > 500?  │     │  │
│  │                          │  ┌────┐           │     │  │
│  │                          │  │HITL│           │     │  │
│  │                          │  └────┘           │     │  │
│  │                          └──────────────────┘     │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

**核心洞察（来自 pi-agent-core）**：
> 所有 Agent 设计模式本质上都是"WorkflowUnit + ControlFlow"的不同组合。
> WorkflowUnit 不感知自己处于何种控制流中（控制流反转），这保证了任意混合模式的逻辑自洽。

## 七、工程选型决策指南

| 你的场景 | 推荐方案 |
|----------|----------|
| 步骤固定、路径明确（如数据清洗、标准 RAG） | 纯 **Workflow** |
| 多业务场景需要智能分流（如客服系统先分类再处理） | **Agentic Workflow**（Router 模式） |
| 复杂任务需先规划再分步执行（如"帮我写一份市场分析报告"） | **Agentic Workflow**（Plan-Execute） |
| 需审批的关键操作嵌入流程（如退款 > 500 元需人工） | **Agentic Workflow**（Human-in-the-Loop） |
| 目标模糊、工具多样、无固定套路（如"帮我订最优差旅方案"） | 完整 **Agent** |
| 大型系统需多专业领域协作 | **Multi-Agent** + 顶层 Agentic 路由 |

---

## 八、总结

1. **Workflow 是骨架**：定义系统的边界和主干路径，保证可靠性
2. **Agentic Workflow 是关节**：在关键分岔口注入智能决策，平衡灵活性与可控性
3. **Agent 是大脑**：在主干无法覆盖的模糊地带做完全自主推理
4. **混合是最佳实践**：固定部分用 Workflow，决策点用 Agentic Workflow，未知地带用 Agent
5. **控制流反转是设计原则**：单元不感知编排方式，才能自由组合
