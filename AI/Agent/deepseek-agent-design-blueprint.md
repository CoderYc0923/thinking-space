# DeepSeek Agent 完整设计蓝图

> 深入理解 AI Agent 的核心架构：从底层原理到全链路工作流

---

## 核心理念

```
Agent = LLM（大脑）+ 工具（手脚）+ 记忆（经验）+ 循环（执行力）+ 安全（护栏）
```

类比到人类解决问题：

| 人类能力 | Agent 对应 | 具体实现 |
|----------|-----------|---------|
| 大脑思考 | LLM 推理 | DeepSeek 大语言模型 |
| 手写代码、眼睛看书 | 工具集 | 代码执行、搜索、文件操作 |
| 短期记忆（草稿纸） | 上下文窗口 | 对话历史管理 |
| 长期记忆（经验） | 持久化记忆系统 | 偏好、事实、对话摘要存储 |
| 做事前想一想安全吗 | 安全层 | 沙箱 + 人机协同确认 |
| 遇到困难换方法 | 反思纠错 | 自动调整策略 |

## 二、核心工作流：感知 → 思考 → 行动 循环

### 主控循环状态机

```
        ┌──────────────────────────────────┐
        │                                  │
        ▼                                  │
   ┌─────────┐     ┌──────────┐     ┌──────────┐
   │ 观察     │────▶│ 思考     │────▶│ 行动     │
   │ Observe │     │ Think    │     │ Act      │
   │         │◀────│          │◀────│          │
   │ 拿结果  │     │ 选工具   │     │ 调工具   │
   └─────────┘     └──────────┘     └──────────┘
        │                                  │
        │           任务完成？              │
        └─────────── 是 ──────────────────▶ 返回最终答案
```

### 每一轮的决策逻辑

```
当前状态 + 用户任务 + 历史上下文 + 可用工具列表
                    ↓
            提交给 LLM 推理
                    ↓
          LLM 返回两种结果之一：
          ┌────────────────┬────────────────┐
          │  Tool Call     │  Final Answer  │
          │  "我需要调用   │  "任务完成，   │
          │   搜索工具"    │   这是结果"    │
          └───────┬────────┴───────┬────────┘
                  │                │
           执行工具调用        返回给用户
           拿到结果后          （循环结束）
           回到循环开头
```

### 时序流程

```
用户 → 主控循环 → 记忆系统 → LLM → 工具集 → 返回结果
 │        │          │        │       │         │
 │  ①输入任务   ②获取上下文  │       │         │
 │        │          │        │       │         │
 │        │    ③返回记忆     │       │         │
 │        │─────────────────▶│       │         │
 │        │    ④提交推理请求  │       │         │
 │        │          │        │       │         │
 │        │    ⑤返回思考+行动决策    │         │
 │        │◀─────────────────│       │         │
 │        │          │        │       │         │
 │        │    ⑥如果需要，执行工具调用│         │
 │        │─────────────────────────▶│         │
 │        │          │        │       │         │
 │        │    ⑦返回工具执行结果     │         │
 │        │◀─────────────────────────│         │
 │        │          │        │       │         │
 │        │    ⑧回到步骤④，继续循环  │         │
 │        │          │        │       │         │
 │  ⑨最终答案     │        │       │         │
 │◀────────│        │        │       │         │
```

## 三、第一层：核心调度层 (Agent OS) 详解

### 3.1 上下文管理器（短期记忆）

**要解决的问题：** LLM 一次能处理的 token 有上限（如 128K），对话长了会超。

**解决方案：**

```
┌─────────────────────────────────────┐
│ 历史消息队列：[msg1][msg2][msg3]...  │
│                 ↓                    │
│ 滑动窗口裁剪：只保留最近 N 轮对话     │
│                 ↓                    │
│ Token 计数器：实时计算当前总 token 数 │
│                 ↓                    │
│ 超限时策略：丢弃最早的消息 / 压缩摘要  │
└─────────────────────────────────────┘
```

**核心策略：**
- **滑动窗口：** 保留最近 N 轮对话，旧的自动丢弃
- **智能裁剪：** 优先保留 System Prompt 和最新的用户消息
- **摘要压缩：** 对过长的历史对话自动生成摘要，用摘要代替原文
- **Token 预算：** 为 System Prompt、工具定义、历史、当前消息各自分配 token 配额

### 3.2 长期记忆系统

**存储内容分类：**

```
┌─────────────────────────────────────────────┐
│           长期记忆系统                       │
│                                             │
│  存储内容：                                 │
│  ├── 用户身份偏好（技术栈、环境、习惯）      │
│  ├── 关键决策记录（选型、架构方案）          │
│  ├── 事实知识（配置路径、API key 等）        │
│  └── 对话摘要（历史对话的压缩版）            │
│                                             │
│  工作原理：                                 │
│  每次新对话开始时 → 检索相关记忆            │
│  对话过程中发现重要信息 → 自动写入记忆       │
│  记忆太多 → 定期清理 / 合并去重             │
└─────────────────────────────────────────────┘
```

**检索策略：**
- **语义检索：** 用当前对话内容做 query，向量搜索相关记忆
- **标签过滤：** 按记忆类型（偏好/事实/决策）筛选
- **时效性排序：** 越新的记忆权重越高
- **关联度评分：** 综合判断哪些记忆对当前任务最有帮助

### 3.3 主控循环 (Agentic Loop) 伪代码

```typescript
async function agenticLoop(userTask: string, tools: Tool[], memory: Memory) {
  // 初始化上下文
  let context = await buildContext(userTask, memory);
  let maxSteps = 20;  // 防止无限循环
  let stepCount = 0;

  while (stepCount < maxSteps) {
    stepCount++;

    // 1. 提交给 LLM 推理
    const response = await llm.chat({
      messages: context.messages,
      tools: tools.map(t => t.definition),  // 传入工具定义
    });

    // 2. 判断 LLM 的决策
    if (response.finishReason === "stop") {
      // 任务完成，返回最终答案
      await memory.save({ type: "conversation_summary", content: response.content });
      return response.content;
    }

    if (response.finishReason === "tool_calls") {
      // LLM 决定调用工具
      for (const toolCall of response.toolCalls) {
        // 3. 安全检查（人机协同）
        if (isSensitiveOperation(toolCall)) {
          const approved = await askUserConfirmation(toolCall);
          if (!approved) continue;  // 用户拒绝，跳过
        }

        // 4. 在安全沙箱中执行工具
        const result = await executeInSandbox(toolCall);

        // 5. 记录日志
        logger.info({ step: stepCount, tool: toolCall.name, params: toolCall.params, result });

        // 6. 将结果追加到上下文
        context.messages.push({
          role: "tool",
          tool_call_id: toolCall.id,
          content: JSON.stringify(result),
        });
      }
      // 回到循环开头，让 LLM 看到工具结果后继续思考
      continue;
    }
  }

  // 超过最大步数，强制终止
  return "任务超时，已达到最大执行步数。";
}
```

## 四、第二层：推理规划层 (大脑) 详解

### 4.1 思维链推理 (Chain of Thought)

让模型显式输出推理过程，而非直接给答案：

```
普通问答：
  问："123 × 456 = ?"
  答："56088"

思维链推理：
  问："123 × 456 = ?"
  思考过程：
    123 × 400 = 49200
    123 × 50  = 6150
    123 × 6   = 738
    49200 + 6150 + 738 = 56088
  答："56088"

Agent 中的应用：
  不是直接给答案，而是让模型先输出思考过程
  → 提高复杂推理的准确率
  → 用户可以看清 Agent 的决策逻辑
  → 出错时更容易定位问题
```

### 4.2 任务分解 (Planning)

将复杂目标自动拆解为多步可执行的子任务：

```
用户任务："帮我做一个电商网站的竞品分析报告"

Agent 自动拆解：
  Step 1: 搜索"2024年电商平台排名"           → 用搜索工具
  Step 2: 对每个竞品搜索"XX平台 优缺点"       → 用搜索工具（批量）
  Step 3: 汇总信息，整理成对比表格            → 用代码解释器
  Step 4: 生成图表可视化                      → 用代码解释器
  Step 5: 输出 Markdown 格式报告              → 用文件系统保存

关键设计：
  - 每步有明确的输入和输出
  - 步骤之间有依赖关系
  - 失败时可以回退或换策略
```

### 4.3 反思纠错 (Reflection)

```
场景：Agent 调用搜索工具，但返回了空结果

❌ 没有反思能力：
  → "搜索没有结果，我无法回答"（直接放弃）

✅ 有反思能力：
  → "关键词可能太具体了，换个宽泛的词试试"
  → "也许可以换一种语言搜索"
  → "这个方向不对，换个思路：直接找行业报告"
  → 自动调整策略，重新尝试

实现方式：
  - 在 Prompt 中加入"如果工具返回错误，请分析原因并尝试替代方案"
  - 设置重试机制，最多尝试 3 次不同策略
  - 记录失败原因，避免重复相同的错误路径
```

### 4.4 工具选择 (Tool Selection)

LLM 根据任务语义，自动判断需要调用哪些工具：

| 用户任务 | LLM 判断 | 选择的工具 |
|---------|---------|-----------|
| "查一下今天北京的天气" | 需要实时数据 | 联网搜索 |
| "计算这个 Excel 的月度汇总" | 需要写代码处理数据 | 代码解释器 |
| "帮我把这封邮件翻译成英文" | 纯语言任务 | 不需要工具，直接翻译 |
| "读取 project/readme.md" | 需要访问文件 | 文件系统工具 |
| "打开淘宝搜索机械键盘" | 需要操控浏览器 | 浏览器操控工具 |
| "检索公司内部的报销政策" | 需要私有知识 | 知识库/RAG 检索 |

## 五、第三层：能力执行层 (工具箱) 详解

### 5.1 工具注册标准 (Function Calling Schema)

每个工具都遵循统一的接口规范：

```typescript
// 工具定义接口
interface ToolDefinition {
  name: string;           // 工具名称，如 "web_search"
  description: string;    // 描述用途，LLM 靠这个判断何时使用
  parameters: {           // 参数 JSON Schema
    type: "object";
    properties: Record<string, {
      type: string;
      description: string;
      enum?: string[];
    }>;
    required: string[];
  };
}

// 工具执行结果
interface ToolResult {
  success: boolean;
  data?: any;
  error?: string;
}

// 完整工具注册示例
const webSearchTool = {
  definition: {
    name: "web_search",
    description: "搜索互联网获取实时信息。当需要查找最新数据、新闻或用户询问的事实时使用此工具。",
    parameters: {
      type: "object",
      properties: {
        query: {
          type: "string",
          description: "搜索关键词，使用中文可获得更好的中文结果"
        },
        topK: {
          type: "integer",
          description: "返回结果数量，默认 5"
        }
      },
      required: ["query"]
    }
  },
  execute: async (params: { query: string; topK?: number }) => {
    // 实际搜索逻辑
    const results = await searchEngine.search(params.query, params.topK || 5);
    return { success: true, data: results };
  }
};
```

### 5.2 核心工具详解

| 工具 | 核心能力 | 典型场景 | 安全要求 |
|-----|---------|---------|---------|
| **代码解释器** | 编写并执行 Python/JS 代码 | 数据分析、图表生成、文件格式转换 | 必须沙箱隔离，限制网络/文件访问，限制执行时间和内存 |
| **联网搜索** | 获取实时信息和最新数据 | 查询新闻、天气、股价、最新技术文档 | 防止 SSRF 攻击，过滤敏感 URL |
| **知识库/RAG** | 检索私有文档和内部知识 | 企业政策查询、产品文档检索 | 权限控制，只返回用户有权查看的内容 |
| **文件系统** | 读取/创建/编辑用户文件 | 读写配置文件、生成报告 | 路径限制在用户目录内，敏感操作需确认 |
| **浏览器操控** | 打开网页、点击、输入、截图 | 网页自动化、信息采集 | 限制可访问域名，防止恶意操作 |
| **自定义 API** | 连接外部服务 | 发送邮件、操作数据库、调用 SaaS API | 鉴权 + 限流，敏感操作需确认 |

### 5.3 工具组合调用

复杂任务往往需要多个工具协同：

```
任务："分析竞品网站的定价策略"

工具调用链：
  1. 浏览器工具 → 打开竞品网站，截图价格页面
  2. 浏览器工具 → 提取页面中的价格数据
  3. 代码解释器 → 用 Python 整理数据、计算均价、生成对比表
  4. 联网搜索 → 搜索行业定价报告作为参考
  5. 代码解释器 → 综合所有数据，生成可视化图表
  6. 文件系统 → 保存分析报告为 PDF
```

### 6.2 人机协同 (Human-in-the-Loop)

## 八、与 LangChain Agent 的对照关系

你在学习 LangChain Agent 开发时，会发现它与这套蓝图完全对应：

| 蓝图层级 | LangChain 对应模块 | 说明 |
|---------|------------------|------|
| 主控循环 (Agentic Loop) | `AgentExecutor` | 执行 "思考→行动→观察" 循环 |
| 工具注册规范 | `Tool` / `StructuredTool` | 定义工具的名称、描述、参数 Schema |
| 上下文管理器 | `BaseChatMessageHistory` | 管理对话历史的存储和裁剪 |
| 长期记忆 | `BaseMemory` / `VectorStoreRetrieverMemory` | 持久化记忆的存储和检索 |
| LLM 推理 | `BaseChatModel` (如 `ChatOpenAI`) | 调用大语言模型进行推理 |
| 工具选择 | `Agent` (如 `OpenAIFunctionsAgent`) | 决定何时调用哪个工具 |
| 思维链 | Prompt 中的 `agent_scratchpad` | 让模型记录中间推理步骤 |

这意味着：**掌握了这套蓝图，你就理解了所有 Agent 框架的核心设计理念。** LangChain、AutoGPT、CrewAI 等框架本质上都是这套蓝图的不同实现方式。

## 九、设计原则总结

```
1. 最小权限原则
   → 只给 Agent 完成当前任务所需的最小权限
   → 能只读就不给写权限，能预览就不直接执行

2. 透明可解释
   → 每一步的思考过程对用户可见
   → 完整的日志记录，出问题可回溯

3. 优雅降级
   → 工具调用失败时不崩溃，而是尝试替代方案
   → 超过最大步数时给出部分结果，而非报错

4. 人机协同
   → 高风险操作必须人类确认
   → 用户随时可以中断 Agent 的执行

5. 持续学习
   → 从成功和失败中学习（记忆系统）
   → 用户偏好自动记录，体验越来越好
```
