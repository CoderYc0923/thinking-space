# AI记账Agent — 完整架构设计文档

> **设计目标**：用户通过自然语言对话实现记账、总结、分析、理财建议等功能，支持单意图、多意图、复合意图输入，最大化降低LLM Token消耗。

---

## 一、架构总览：四大模块 × 三层模型

本架构采用当前业界主流的 **"四大模块 + 感知-决策-执行三层模型"** 设计范式：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        AI记账Agent 整体架构                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐     │
│   │  🧠 大脑  │    │  📋 规划  │    │  🧠 记忆  │    │  🔧 工具  │     │
│   │  (LLM)   │◄──►│(Orchestr)│◄──►│(Memory)  │◄──►│(Tools)   │     │
│   └────┬─────┘    └────┬─────┘    └────┬─────┘    └────┬─────┘     │
│        │               │               │               │           │
│   ┌────▼───────────────▼───────────────▼───────────────▼─────┐     │
│   │                    三层处理流水线                          │     │
│   │  ┌──────────────┬──────────────────┬──────────────────┐  │     │
│   │  │  感知层       │     决策层        │     执行层        │  │     │
│   │  │ (Perception)  │  (Decision)       │  (Execution)      │  │     │
│   │  └──────────────┴──────────────────┴──────────────────┘  │     │
│   └──────────────────────────────────────────────────────────┘     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.1 四大模块

| 模块 | 职责 | 核心组件 |
|------|------|----------|
| **🧠 大脑 (Brain)** | 理解用户意图、生成响应、复杂推理 | 分层LLM（L0规则→L1轻量→L2标准→L3强模型） |
| **📋 规划 (Planning)** | 任务分解、依赖排序、并行调度 | Orchestrator（DAG编排引擎） |
| **🧠 记忆 (Memory)** | 上下文管理、历史检索、用户画像 | 三层记忆（工作/短期/长期）+ 向量数据库 |
| **🔧 工具 (Tools)** | 执行具体业务操作 | 记账/查询/统计/建议四类工具集 |

### 1.2 三层处理模型

| 层次 | 职责 | 对应模块 | Token消耗 |
|------|------|----------|-----------|
| **感知层 (Perception)** | 意图识别、实体提取、多意图拆分 | 大脑（意图分类）+ 记忆（上下文检索） | 0-100T |
| **决策层 (Decision)** | 任务编排、依赖分析、模型选择 | 规划（Orchestrator）+ 大脑（模型路由） | 0-200T |
| **执行层 (Execution)** | 工具调用、结果聚合、响应生成 | 工具（Function Calling）+ 大脑（响应生成） | 0-500T |

## 二、主流方案对标分析

### 2.1 业界主流Agent架构范式

当前（2025-2026）业界主流Agent架构普遍采用以下范式：

| 范式 | 代表框架 | 核心思想 | 本架构对应 |
|------|----------|----------|------------|
| **ReAct** | LangChain | 思考→行动→观察循环 | 决策层核心循环 |
| **Plan-and-Execute** | LangGraph | 先规划再执行 | 规划模块DAG编排 |
| **Multi-Agent** | AutoGen, CrewAI | 多Agent协作 | 可扩展为子Agent模式 |
| **分层路由** | LiteLLM, Portkey | 按复杂度路由不同模型 | 大脑分层LLM |
| **RAG增强** | LlamaIndex | 检索增强生成 | 记忆模块向量检索 |

### 2.2 "四大模块+三层"是否为当前主流？

**是的，这是当前最主流的设计范式。** 理由如下：

1. **四大模块划分** 源自Lilian Weng（OpenAI）2023年提出的Agent核心组件理论，被后续几乎所有框架继承：
   - 大脑 = LLM推理核心
   - 规划 = 任务分解与反思
   - 记忆 = 短期/长期记忆管理
   - 工具 = 外部能力扩展

2. **感知-决策-执行三层** 是认知架构在Agent领域的映射，被阿里云2025年《AI Agent架构新范式报告》、Google DeepMind的Agent架构设计等广泛采用。

3. **2026年趋势**：Multi-Agent协作 + MCP协议 + A2A协议成为新热点，但底层仍然遵循四大模块+三层模型的基本范式。

4. **本架构的优势**：在遵循主流范式的基础上，针对记账场景做了深度优化——特别是Token消耗控制，这是通用框架通常忽略的。

**多意图处理示例：**

```
输入: "午餐花了35元，顺便记一下咖啡28块，对了这个月餐饮花了多少？
      感觉有点多了，帮我设个2000的预算吧"

拆分结果:
┌──────────────────────────────────────────────┐
│ Task 1: add_transaction(餐饮, 35, "午餐")     │ priority=1, 无依赖
│ Task 2: add_transaction(餐饮, 28, "咖啡")     │ priority=1, 无依赖
│ Task 3: query(分类=餐饮, 时段=本月)            │ priority=2, 无依赖
│ Task 4: set_budget(分类=餐饮, 金额=2000)      │ priority=3, 依赖Task3结果
└──────────────────────────────────────────────┘

DAG:
  [Task1] ──┐
             ├──→ [Task3] ──→ [Task4]
  [Task2] ──┘

执行:
  T0: Task1 ∥ Task2 (并行)
  T1: Task3
  T2: Task4 (使用Task3的数据做判断)
  T3: 聚合结果 → 生成综合回复
```

## 四、Token消耗优化策略

### 4.1 优化策略总览

| 策略 | 原理 | 节省 | 难度 |
|------|------|------|------|
| **规则引擎优先** | Regex直接提取参数 | 40-50%请求0T | ⭐⭐ |
| **语义缓存** | 相似请求复用结果 | 15-20%命中 | ⭐⭐⭐ |
| **分层模型** | 简单任务用小模型 | 成本降低60-80% | ⭐⭐ |
| **System Prompt缓存** | 固定部分用Cache API | 每次省~200T | ⭐ |
| **工具定义缓存** | Function definitions缓存 | 每次省~200T | ⭐ |
| **上下文压缩** | 历史摘要替代完整对话 | 长对话省50-70% | ⭐⭐⭐ |
| **结构化输出** | JSON Mode减少废话 | 输出省30-50% | ⭐ |
| **本地分类器** | BERT替代LLM分类 | 分类T归零 | ⭐⭐⭐⭐ |
| **预计算统计** | 定时任务预聚合 | 查询0T | ⭐⭐ |
| **模板响应** | 高频回复预定义 | 生成T归零 | ⭐ |

### 4.2 成本估算

## 七、设计原则总结

| # | 原则 | 说明 |
|---|------|------|
| 1 | **分层LLM** | 简单操作用小模型，复杂推理用大模型 |
| 2 | **意图前置** | 先路由再处理，避免无效Token消耗 |
| 3 | **本地优先** | 能本地处理的绝不调用LLM |
| 4 | **缓存为王** | 语义缓存 + 结果缓存 + 模板缓存 |
| 5 | **结构化输出** | JSON Mode减少解析Token |
| 6 | **增量上下文** | 按需加载历史，非全量注入 |
| 7 | **无状态工具** | 工具函数纯函数化，便于并行和测试 |
| 8 | **模型降级** | L0→L1→L2→L3逐级升级，不越级 |

> **核心思想：能用代码解决的绝不用模型，能用小模型解决的绝不用大模型。**
