# 18. 异常体系

## 18.1 层级

```plaintext
java.lang.Throwable  // 所有异常/错误的顶层父类
├─ Error：系统级严重错误，程序无力处理
│  ├─ OutOfMemoryError 内存溢出
│  ├─ StackOverflowError 栈溢出
│  └─ NoClassDefFoundError 类找不到
└─ Exception：程序可捕获、可处理的异常
   ├─ RuntimeException 运行时异常（非受检异常，无需强制捕获）
   │  ├─ NullPointerException 空指针
   │  ├─ ArrayIndexOutOfBoundsException 数组越界
   │  ├─ ClassCastException 类型强转异常
   │  ├─ ArithmeticException 算术异常（除0）
   │  └─ IllegalArgumentException 参数非法
   └─ 受检异常（CheckedException）编译强制处理（强制 try-catch 或 throws 抛出）
      ├─ IOException IO读写异常
      ├─ SQLException 数据库异常
      ├─ FileNotFoundException 文件不存在
      └─ ParseException 日期解析异常
```

1. Error错误

   - 定义：JVM 运行时严重故障，**代码无法修复、无法捕获处理**
   - 产生原因：硬件资源不足、JVM 崩溃、内存耗尽
   - 特点：程序只能终止，不需要 try-catch
   - 常见：OOM、栈溢出、类加载失败

2. Exception 异常（开发重点）

   - 运行时异常 RuntimeException（非受检异常）

     - 编译期**不强制捕获、不强制抛出**

     - 根源是**代码编写错误**，规范编码就能避免

     - 运行时才抛出

     - 举例：空指针、数组越界、除 0、类型转换错误

     - ```java
       // 空指针，编辑不报红，运行报错
       String str = null;
       str.length();
       ```

     - 

   - 受检异常 CheckedException

     - 编译期强制处理，不捕获 / 不抛出直接编译报错
     - 根源是**外部资源不可控**（文件、网络、数据库）
     - 必须二选一: `try-catch` 捕获处理\方法声明 `throws` 抛出给上层

     ```java
     // 文件读取，编译强制处理
     // 报错：未处理受检异常IOException
     FileReader fr = new FileReader("test.txt");
     ```

     

## 18.2 处理语法

1. try-catch-finally：finally 无论是否异常都会执行，用于释放资源

   - ```java
     try {
         // 可能抛出异常的代码
     } catch (异常类型1 e) {
         // 处理对应异常
     } catch (异常类型2 e) {
         // 多异常分层捕获，先子类后父类
     } finally {
         // 无论是否异常、是否return，一定会执行（唯一例外：System.exit()）
     }
     
     //示例
     try {
         FileReader fr = new FileReader("a.txt");
         int num = 1 / 0;
     } catch (FileNotFoundException e) {
         System.out.println("文件不存在");
     } catch (ArithmeticException e) {
         System.out.println("除数不能为0");
     } catch (Exception e) {
         // 兜底，捕获所有其他异常
         e.printStackTrace();
     } finally {
         System.out.println("资源释放，一定会执行");
     }
     ```

   - finally 核心用途：释放 IO 流、数据库连接、锁等资源。Java7 推出 `try-with-resources` 简化资源关闭

2. try-with-resources（Java7 推出）：自动关闭实现 AutoCloseable 的 IO 资源，自动执行 close ()，不用手写 finally

   - ```java
     // 不用手动fr.close()
     try (FileReader fr = new FileReader("test.txt")) {
         char c = (char) fr.read();
     } catch (IOException e) {
         e.printStackTrace();
     }
     ```

   - 

3. throw：手动抛出异常（方法内部主动抛）

   - ```java
     public void setAge(Integer age) {
         if (age < 0 || age > 150) {
             // 主动抛出运行时异常，无需throws声明
             throw new IllegalArgumentException("年龄非法");
         }
     }
     ```

   - 

4. throws：方法声明抛出异常，交给上层处理

   - 用于**受检异常**，写在方法签名后，告诉上层调用方：我这里可能抛异常，你自己处理

   - ```java
     // IO是受检异常，必须throws
     public void readFile() throws IOException {
         FileReader fr = new FileReader("1.txt");
     }
     ```

## 18.4 异常对象常用方法

```java
try {
    int a = 1 / 0;
} catch (ArithmeticException e) {
    e.getMessage();    // 获取异常简短描述：/ by zero
    e.toString();      // 异常类型+描述
    e.printStackTrace(); // 打印完整堆栈信息（开发调试用）
}
```

## 18.5 自定义业务异常（SpringBoot 实战必备）

```java
//系统内置异常信息模糊，前端需要统一错误码、错误提示，自定义业务异常统一处理
public class BusinessException extends RuntimeException {
    // 错误码
    private Integer code;
    // 错误信息
    private String msg;
    
    public BusinessException(Integer code, String msg) {
        super(msg);
        this.code = code;
        this.msg = msg;
    }
    
    public Integer getCode() { return code; }
    public Integer getMsg() { return msg; }
}


//业务代码抛出
User user = userMapper.selectById(id);
if (user == null) {
    throw new BusinessException(4001, "用户信息不存在");
}



// 全局异常处理器 @RestControllerAdvice
// 统一捕获所有 Controller 异常，返回标准化 JSON，不用每个接口写 catch
@RestControllerAdvice
public class GlobalExceptionHandler {
    // 捕获自定义业务异常
    @ExceptionHandler(BusinessException.class)
    public Result<?> handleBusinessEx(BusinessException e) {
        return Result.fail(e.getCode(), e.getMsg());
    }
    
    // 兜底捕获所有未知异常
    @ExceptionHandler(Exception.class)
    public Result<?> handleAllEx(Exception e) {
        log.error("系统异常", e);
        return Result.fail(500, "服务器内部错误");
    }
}


```

## 18.6 异常开发规范 & 避坑点

1. 捕获异常不要只打印 e.printStackTrace ()

   - 线上环境日志无法留存堆栈，替换为日志框架记录：

     ```java
     log.error("读取文件失败", e);
     ```

2. 不要捕获顶级 Exception 偷懒

   - 分层捕获，精准处理对应异常，不要一把抓所有异常，隐藏未知 bug。

3. finally 中不要写 return

   - 会覆盖 try/catch 里的 return，丢失正确返回值。

4. 不要吞异常（空 catch）

   ```java
   // 禁止！异常消失，无法定位问题
   try {
       readFile();
   } catch (IOException e) {
       // 空代码，吞掉异常
   }
   ```

5. 区分运行时 / 受检异常

   - 业务校验类异常用**自定义 RuntimeException**，不用写 throws，简化代码
   - IO、网络、文件操作必须处理受检异常

6. 异常不要用于正常业务流程

   - 异常创建、堆栈打印性能差，频繁抛出会严重影响接口速度
   - 能用 if 判断拦截的逻辑，不要靠异常兜底

## 18.7 面试题汇总

1. Error 和 Exception 的区别？
   - Error 是JVM致命错误无法处理
   - Exception 是程序可捕获恢复的异常
2. 运行时异常和受检异常区别？
   - 运行时异常编译不强制处理，由代码错误导致
   - 受检异常编译强制捕获/抛出，外部资源问题（数据库，网络，文件等）
3. finally 一定会执行吗？
   - 不一定，当System.exit(0)直接终止JVM
4. throw 和 throws 区别？
   - throw在方法内部手动抛出异常对象
   - throws在方法声明，声明该方法可能抛出异常
5. try-with-resources 原理？
   - 资源类实现AutoCloseable，代码执行完毕自动调用close()，自动释放资源
6. 为什么自定义业务异常一般继承 RuntimeException？
   - 因为不需要在方法上写throws，上层调用不强制捕获，配合全局异常处理器统一处理
7. 什么是异常链？·
   - 构造异常时，传入原始异常，保留完整堆栈，方便定位根因
