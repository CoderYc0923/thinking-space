# 19. 多线程（对比 JS 单线程）

JS：单线程事件循环，异步不新开线程；Java 真正多线程并发，存在线程安全竞争。

进程：操作系统资源分配最小单位，独立内存、CPU资源，互相隔离。比如浏览器、IDEA、JVM

线程：进程内独立执行路径，CPU调度最小单位；共享进程堆/方法区资源，私有栈、程序计数器。

一个进程至少有一个主线程，JVM启动默认main主线程

并发：单核CPU，时间片快速切换多个线程，宏观同时运行

并行：多核CPU，多个线程真正同时执行

## 19.1 线程的 5 种生命周期状态

- Thread.State
  - NEW 新建
    - new Thread() 创建对象，未调用start()
  - RUNNABLE 就绪/运行
    - 调用start()，等待CPU时间片；拿到时间片就运行run()
  - BLOCKED 阻塞
    - 等待synchronized锁，锁释放后回到RUNNABEL
  - WAITING 无限等待
    - wait() / join()，无超时，需要等待其他线程唤醒
  - TIMED_WAITING 限时等待
    - sleep() / wait()，超时自动唤醒
  - TREMINATED 终止
    - run方法执行完毕/异常退出，线程销毁，不可重启
- 关键：start()只能执行一次，重复调用抛IllegalThreadStateException

## 19.2 四种创建线程方式

1. 实现 Runnable 接口（推荐，无单继承限制）

   - 无返回值，适合单纯任务

   - ```java
     // 定义任务
     class Task1 implements Runnable {
         @Override
         public void run() {
             System.out.println("子线程执行：" + Thread.currentThread().getName());
         }
     }
     
     // 启动任务
     public class Test {
         public static void main(String[] args){
             Thread t1 = new Thread(new Task1(), "线程1");
             t1.start();
         }
     }
     
     ```

   - 

2. 继承 Thread 类（不推荐，占用继承位）

   - ```java
     class Thread1 extends Thread {
         @Override
         public void run() {
             System.out.println("继承Thread");
         }
     }
     
     // use
     new Thread1().start();
     ```

   - 

3. 实现 Callable+Future：带返回值、可抛异常

   - ```java
     // Runnable 无返回值，Callable 可获取执行结果
     import java.util.concurrent.Callable;
     import java.util.concurrent.FutureTask;
     
     class MyCall implements Callable<Integer> {
         @Override
         public Integer call() throws Exception {
             int sum = 0;
             for(int i=1;i<=10;i++) sum += i;
             return sum;
         }
     }
     
     public class Test {
         public static void main(String[] args) throws Exception {
             FutureTask<Integer> task = new FutureTask<>(new MyCall());
             Thread t1 = new Thread(task);
             t.start();
             // 阻塞获取返回值
             Integer res = task.get();
             System.out.println(res);
         }
     }
     ```

4. 线程池（生产环境唯一推荐）

   - 频繁创建销毁线程开销极大，复用线程，下文单独详解。


## 19.3 线程常用核心方法

1. 静态方法（Thread.xxx，作用当前执行线程）

   - Thread.sleep(long ms)
     - 让当前线程限时休眠，**不释放锁**；抛出受检异常必须捕获 /throws
   - Thread.currentThread()
     - 获取当前运行线程对象
   - Thread.yield()
     - 让出 CPU 时间片，回到就绪态，不保证一定让出

   

2. 对象实例方法

   - start()

     - 启动线程，底层调用 native 方法创建系统线程

   - run()

     - 线程执行逻辑，**直接调用 run 只是普通方法，不会新开线程**

   - join()

     - 等待该线程执行完毕，当前线程阻塞；底层调用 wait ()，释放锁。类似js await

     - ```java
       Thread t = new Thread(() -> {for(int i=0;i<5;i++) System.out.println(i);});
       t.start();
       t.join(); // main线程等待t执行完才继续往下走
       System.out.println("执行结束");
       ```

     - 

   - setName()/getName()

     - 设置 / 获取线程名

   - setPriority()

     - 优先级 1~10，默认 5，仅为调度提示，不保证生效

   - interrupt()

     - 标记线程中断标志
       - 阻塞线程 (sleep/wait/join) 会抛出`InterruptedException`
       - 运行中线程仅修改标记，不会停止

## 19.4 线程安全问题

定义：

1. 线程安全指的是多个线程并发访问同一对象/资源时，无需额外同步即可保证执行结果正确且数据一致。
2. 线程不完全则指缺乏保护机制时，并发访问（尤其写操作）会导致数据竞争、结果错误或状态混乱。
3. 判断关键：是否依赖外部同步；若去掉所有锁/同步代码后结果仍然正确，则为线程安全。

产生安全问题主要原因：

1. 非原子操作：如i++实际包括“读改写”三步，线程切换会导致更新丢失
2. 内存可见性缺失：多线程共享**全局变量 / 成员变量**，一线程修改变量后，其他线程因缓存未刷新而读到旧值
3. 指令重排序：线程切换，执行中途让出 CPU，编译器/JVM优化改变执行顺序，破坏逻辑依赖

典型示例对比：

1. 不安全：ArrayList、HashMap、StringBuilder（多线程直接修改会丢数据或抛异常）

2. 安全：String（不可变）、ConcurrentHashMap（内部锁/CAS保护）、StringBuffer（方法级同步）

3. 场景演示：两个线程各自增5万次共享计数器，若无同步，最终结果常小于10万；加synchronized或使用AtomicInteger后可保证正确。

   ```java
   // num++非原子操作
   class Count {
       int num = 0;
       public void add() { num++; }
   }
   
   public class Test {
       public static void main(String[] args) throws InterruputedException {
           Count c = new Count();
           
           // 100个线程各加1000次，预期100000，实际远小于
           for(int i=0;i<100;i++) {
               new Thread(() -> {
                   for(int j=0;j<1000;j++) c.add();
               }).start();
           }
           Thread.sleep(2000);
           System.out.println(c.num);
       }
   }
   ```

   解决方案1:synchronized 同步锁（内置锁，最简单）

   ```java
   //给方法加锁，同一时间只能一个线程执行 add，保证整个num++原子执行
   //优点：写法极简，不用手动释放锁；异常自动释放锁，不会死锁
   //缺点：锁粒度大，并发性能一般；不支持公平锁、超时获取锁等高级功能
   class Count {
       int num = 0;
       // 同步方法，锁对象this
       public synchronized void add() { num++; }
   }
   
   // 进阶 同步代码块（缩小锁范围，性能更好）
   class Count {
       int num = 0;
       Object lock = new Object(); //自定义锁对象
       public void add() { 
       	synchronized (lock) {
               num++;
           }
       }
   }
   ```

   解决方案2：ReentrantLock 显式锁（灵活可控）

   ```java
   //手动加锁、手动释放，功能更强
   //优点: 支持公平锁、非公平锁;支持tryLock()超时获取，避免死锁;可中断锁、多条件变量
   //缺点:代码稍繁琐，忘记 unlock 会永久死锁
   import java.util.concurrent.locks.Lock;
   import java.util.concurrent.locks.ReentrantLock;
   
   class Count {
       int num = 0;
       Lock lock = new ReentrantLock();
       public void add() { 
       	lock.lock();
           try {
               nums++;
           } finally {
               lock.unlock(); // 必须finally释放，防止异常死锁
           }
       }
   }
   ```

   解决方案3:AtomicInteger 原子类（无锁，高性能，推荐数值自增场景）

   ```java
   //底层 CAS 硬件指令，不使用操作系统锁，并发性能最高
   //优点:无锁机制，并发性能远高于 synchronized/Lock;代码简洁，专门用于数字增减场景
   //缺点: 只适合单一数值原子操作，复杂业务逻辑（多变量同时修改）无法使用
   import java.util.concurrent.atomic.AtomicInteger;
   
   class Count {
       AtomicInteger num = new AtomicInteger(0);
       public void add() { 
       	num.incrementAndGet(); // 原子自增，一步完成
       }
   }
   ```

   解决方案4:补充方案：ThreadLocal（数据隔离，特殊场景）

   ```java
   // 不锁资源，每个线程拥有独立变量副本，不存在共享竞争
   // 适合：每个线程独立计数，不需要汇总全局总数
   // 如果需要全局总和，此方案不适用
   class Count {
       ThreadLocal<Integer> num = ThreadLocal.withInitial(() -> 0);
       public void add() { 
       	num.set(num.get() + 1);	
       }
   }
   ```

## 19.5 等待唤醒机制

仅能在`synchronized`同步代码块内调用，作用于锁对象

1. obj.wait()
   - 释放锁，进入 WAITING 等待队列
2. obj.notify()
   - 随机唤醒一个等待线程
3. obj.notifyAll()
   - 唤醒全部等待线程（推荐，避免线程丢失）

生产者消费者极简案例

```java
class Warehouse {
    int stock = 0;
    public synchronized void produce() throws InterruputException {
        if (stock >= 10) wait();// 满了等待
        stock++;
        System.out.println("生产，库存：" + stock);
        notifyAll();
    }
    
    public synchronized void consume() throws InterruputException{
        if (stock <= 0) wait(); //无货等待
		stock--;
        System.out.println("消费，库存：" + stock);
        notifyAll();
    }
}


```



## 19.6 volatile 关键字（轻量级同步）

1. 作用

   - **保证可见性**：修改后立即刷新主存，其他线程读取最新值
   - 禁止指令重排
   - **不保证原子性**：不能解决`count++`自增安全问题

2. 适用场景：状态标记（开关变量）

   ```java
   volatile boolean flag = true;
   new Thread(() -> {
       while(flag) {}
       System.out.println("线程结束");
   }).start();
   // 主线程修改，子线程立刻可见
   flag = false;
   ```

   

## 19.8 线程池

1. 为什么用线程池

   - 复用线程，减少创建销毁开销
   - 控制并发数量，防止无限创建线程导致 OOM
   - 提供延迟、定时、任务拒绝策略

2. 核心五大参数（ThreadPoolExecutor）

   - ```java
     public ThreadPoolExecutor(
     	int corePoolSize, // 核心线程数（常驻不回收）
         int maximumPoolSize, // 最大线程数
         long keepAliveTime, // 非核心线程空闲存活时间
         TimeUnit unit,  // 时间单位
         BlockingQueue<Runnable> workQueue, // 阻塞队列
         ThreadFactory threadFactory, // 线程工厂（命名）
         RejectedExecutionHandler handler // 拒绝策略
     )
     ```

3. 四种拒绝策略（任务满了无法处理）

   - AbortPolicy 默认，直接抛异常（生产推荐捕获处理）
   - DiscardPolicy 丢弃新任务，无提示（丢失数据，慎用）
   - DiscardOldestPolicy 丢弃队列最老任务
   - CallerRunsPolicy 交给提交任务的主线程执行（限流）

4. Executors 工具类（不推荐生产使用，易 OOM）

   - `newFixedThreadPool(n)`：固定线程数，无界队列，任务堆积 OOM
   - `newSingleThreadExecutor()`：单线程池
   - `newCachedThreadPool()`：可缓存线程，最大无上限，大量并发 OOM
   - `newScheduledThreadPool()`：定时延迟线程池

5. 标准业务线程池示例

   ```java
   import java.util.concurrent.*;
   
   public class ThreadPoolUtil {
       // 全局单例线程池
       public static ExecutorService getPool() {
           return new ThreadPoolExecutor(
           	10,
                   50,
                   60L,
                   TimeUnit.SECONDS,
                   new ArrayBlockingQueue<>(200),
                   Executors.defaultThreadFactory(),
                   new ThreadPoolExecutor.CallerRunsPolicy()
           );
       }
   }
   
   //使用
   ExecutorService pool = ThreadPoolUtil.getPool();
   pool.submit((() -> System.out.println("执行任务")));
   pool.shutdown(); //不再接收新任务，等待已有任务执行完毕关闭
   ```

   

## 19.9 死锁

1. 产生四个必要条件（同时满足才会死锁）

   - 互斥：锁同一时间只能一个线程持有
   - 持有并等待：线程持有锁，同时申请新锁、不释放旧锁
   - 不可剥夺：锁只能持有者主动释放，其他线程不能强制抢走
   - 循环等待：线程形成环形锁依赖（线程 1 拿 A 等 B，线程 2 拿 B 等 A）

   示例

   ```java
   Object lockA = new Object();
   Object lockB = new Object();
   
   // 线程1
   new Thread(() -> {
       synchronized (lockA) {
           Thread.sleep(100);
           synchronized (lockB) {}
       }
   }).start();
   
   // 线程2
   new Thread(() -> {
       synchronized (lockB) {
           Thread.sleep(100);
           synchronized (lockA) {}
       }
   }).start();
   ```

   解决方案1：统一锁获取顺序（破坏循环等待）（最简单、推荐）

   ```java
   // 核心：所有线程按照固定顺序获取锁，消除循环等待的条件
   // 示例：规定必须先获取A锁，在获取B锁
   // 优点：改动极小，性能无损耗，业务首选。
   public class FixOrderDemo {
       static Object lockA = new Object();
       static Object lockB = new Object();
       
       public static void main(String[] args) {
           //线程1
           new Thread(() -> {
               synchronized (lockA) {
                   System.out.println("线程1 获取 lockA");
                   try {
                       Thread.sleep(100);
                   } catch (InterruputException e){
                       e.printStackTrace();
                   }
                   
                   synchronized (lockB) {
                       System.out.println("线程1 获取 lockB");
   
                   }
               }
           },"线程1").start();
           
           // 线程2： 也先拿A锁，再拿B锁，顺序统一
           new Thread(() -> {
               synchronized (lockA) {
                   System.out.println("线程2 获取 lockA");
                   try {
                       Thread.sleep(100);
                   } catch (InterruptedException e) {
                       e.printStackTrace();
                   }
                   synchronized (lockB) {
                       System.out.println("线程2 获取 lockB");
                   }
               }
           }, "线程2").start();
       }
   }
   ```

   

   解决方案2：使用ReentrantLock.tryLock 限时抢锁（破坏持有并等待）

   ```java
   // 核心: 获取第二把锁时设置超时时间，拿不到就主动释放已持有的锁，不一直阻塞等待
   
   //优点：适合无法统一锁顺序的复杂业务；超时后主动释放锁，不会永久阻塞。
   
   import java.util.concurrent.locks.Lock;
   import java.util.concurrent.locks.ReentrantLock;
   
   public class TryLockDemo {
       static Lock lockA = new ReentrantLock();
       static Lock lockB = new ReentrantLock();
       
       public static void main(String[] args) {
           new Thread(() -> {
               boolean getA = lockA.tryLock();
               
               if (getA) {
                   try {
                       System.out.println("线程1 获取 lockA");
                       Thread.sleep(100);
                       //限时200毫秒尝试获取B锁
                       boolean getB = lockB.tryLock(200, java.util.concurrent.TimeUint.MILLISECONDS);
                       if (getB) {
                           try {
                              System.out.println("线程1 获取 lockB"); 
                           } finally {
                               lockB.unLock();
                           }
                       } else {
                           // 超时拿不到B，释放A
                           System.out.println("线程1 获取lockB超时，释放lockA");
                       }
                   } catch (InterruputExcep e) {
                       e.printStackTrace();
                   } finally {
                       lockA.unLock();
                   }
               }
           }, "线程1").start();
           
           // 线程2 同样使用tryLock限时获取
           new Thread(() -> {
               boolean getB = lockB.tryLock();
               if (getB) {
                   try {
                       System.out.println("线程2 获取 lockB");
                       Thread.sleep(100);
                       boolean getA = lockA.tryLock(200, java.util.concurrent.TimeUnit.MILLISECONDS);
                       if (getA) {
                           try {
                               System.out.println("线程2 获取 lockA");
                           } finally {
                               lockA.unlock();
                           }
                       } else {
                           System.out.println("线程2 获取lockA超时，释放lockB");
                       }
                   } catch (InterruptedException e) {
                       e.printStackTrace();
                   } finally {
                       lockB.unlock();
                   }
               }
           }, "线程2").start();
       }
   }
   ```

   

   解决方案3：减少锁嵌套粒度（破坏持有并等待）

   ```java
   // 核心：尽量不嵌套同步块，拿到第一把锁执行完逻辑立刻释放，再去获取第二把锁，不 “持有一把等另一把”
   
   //局限：如果业务要求必须同时持有两把锁完成原子操作，则无法使用。
   
   public class LessNestDemo {
       static Object lockA = new Object();
       static Object lockB = new Object();
       
       public static void main(String[] args) {
           new Thread(() -> {
               // 先单独操作A锁，释放了再操作B锁
               synchronized (lockA) {
                   System.out.println("线程1 操作lockA");
               }
               try {
                   Thread.sleep(100);
               } catch (InterruptedException e) {
                   e.printStackTrace();
               }
               synchronized (lockB) {
                   System.out.println("线程1 操作lockB");
               }
           }, "线程1").start();
           new Thread(() -> {
               synchronized (lockB) {
                   System.out.println("线程2 操作lockB");
               }
               try {
                   Thread.sleep(100);
               } catch (InterruptedException e) {
                   e.printStackTrace();
               }
               synchronized (lockA) {
                   System.out.println("线程2 操作lockA");
               }
           }, "线程2").start();
       }
   }
   ```

   

   解决方案4：统一合并为一把锁（消除多锁竞争）

   ```java
   // 如果业务允许，直接只用单个锁对象，不存在多锁循环等待。
   // 缺点：并发粒度太大，同一时间只能一个线程执行，并发性能最差，仅低并发简单场景使用
   
   public class SingleLockDemo {
       // 只使用一把全局锁
       static Object lock = new Object();
       
       public static void main(String[] args) {
           new Thread(() -> {
               synchronized (lock) {
                   System.out.println("线程1 执行全部逻辑");
                   Thread.sleep(200);
               }
           }, "线程1").start();
           new Thread(() -> {
               synchronized (lock) {
                   System.out.println("线程2 执行全部逻辑");
                   Thread.sleep(200);
               }
           }, "线程2").start();
       }
   }
   ```

   

2. 避免死锁方案

   - 统一锁获取顺序（破坏循环等待）
   - 使用 tryLock 限时抢锁 （破坏持有并等待）
   - 减少同步代码块粒度，不嵌套锁 （破坏持有并等待）

## 19.10 高频面试题总结

1. 进程和线程区别？
   - 进程是操作系统资源分配最小单位，进程间资源隔离；
   - 线程是进程内独立执行路径，CPU调度最小单位，共享进程资源，切换开销小
2. run () 和 start () 区别？
   - start是启动线程，底层调用native方法创建系统线程，只能调用一次，重复调用抛出异常IllegalThreadStateException
   - run是普通业务方法，线程执行逻辑，直接执行不会新开线程
3. sleep 和 wait 区别？
   - sleep是线程的静态方法，让当前线程休眠，不释放锁。
   - wait是Object方法，释放锁，需同步块。
4. synchronized 和 Lock 区别？
   - synchronized 同步锁，内置锁，自动释放。
   - Lock手动解锁，支持公平锁，超时、中断。
5. volatile 能否保证原子性？
   - 不能，进保证可见性、禁止重排；自增操作仍需锁 / 原子类
6. 线程池核心参数？拒绝策略有哪些？
   - 核心线程数，最大线程数，空闲时间，阻塞队列，线程工厂，拒绝策略
   - AbortPolicy 默认，直接抛出异常
   - DiscardPolicy 丢弃新任务，无提示
   - DiscardOldestPolicy 丢弃队列最老任务
   - CallerRunsPolicy 交给提交任务的主线程执行（限流）
7. 为什么不推荐 Executors 创建线程池？
   - 底层无界阻塞队列 / 无限最大线程，高并发任务堆积会导致OOM
8. 多线程安全解决方案？
   - synchronized、Lock锁、Atomic原子类、ThreadLocal、不可变对象。
9. 什么是死锁，如何避免？
   - 同时满足：互斥、持有并等待、不可剥夺、循环等待时，会产生死锁
   - 破坏循环等待（统一锁顺序），破坏持有并等待（限时锁，减少嵌套同步）
10. ThreadLocal 作用？
    - 让线程私有存储，每个线程独立副本，隔离多线程数据。常用于登录上下文、数据源切换



## 19.11 补充

#### 19.11.1 ThreadLocal

1. 作用

   - 提供线程私有存储：每个线程拥有独立的数据副本，多线程之间数据完全隔离，天然不存在线程安全竞争，不需要加锁。

2. 原理

   - 存储载体

     - Thread内部ThreadLocalMap

     - ```java
       ThreadLocal.ThreadLocalMap threadLocals = null;
       ```

     - 数据不存放在ThreadLocal对象里

     - 数据存在线程自己的threadLocals（ThreadLocalMap）

     - ThreadLocal只是操作工具，提供set/get/remove方法，用来读写当前线程的map

   - ThreadLocalMap 结构

     - 底层是自定义哈希表，Entry数组

     - ```java
       static class Entry extends WeakReference<ThreadLocal<?>> {
           Object value;
           Entry(ThreadLocal<?> k, Object v) {
               super(k);
               value = v;
           }
       }
       ```

     - 关键

       - key = ThreadLocal 对象（弱引用WeakReference）
       - value = 我们存入的业务数据

     - 为什么key设为弱引用

       - 若ThreadLocal对象被外部强引用置空t1 = null
       - 弱引用key会被垃圾回收自动回收
       - 避免线程长期存活（线程池复用线程）导致key内存泄漏：（只是key是弱引用，value是强引用，不手动remove依然会内存泄漏）

   - set()完整执行流程

     - ```java
       ThreadLocal<String> tl = new ThreadLocal<>();
       tl.set("用户信息");
       // 执行流程
       // 1 获取当前执行线程 Thread t = Thread.currentThread();
       
       //2 获取线程内部的ThreadLocalMap map = t.threadLocals;
       
       //3 如果map为空，新建ThreadLocalMap;
       
       //4 以当前ThreadLocal对象为key,业务数据为value，存入Entry
       
       //5 哈希冲突处理：线性探测法（和HashMap 链表/红黑树不同），向后遍历数组找空位
       ```

   - get()流程

     - ```java
       // 1 获取当前线程、线程的ThreadLocalMap;
       // 2 以当前ThreadLocal为key，查询Entry
       // 3 找到则返回Entry的value
       // 4 找不到返回初始值 initialValue()
       ```

   - remove()流程

     - ```java
       // 1 获取当前线程的map
       // 2 根据ThreadLocal key删除对应的Entry
       // 3 开发规范：使用完必须remove()，防止内存泄漏，因为value还是强引用
       ```

   - initalValue()流程

     - ```java
       // 重写该方法，线程第一次get无数据时自动初始化
       ThreadLocal<Ingeter> tl = new ThreadLocal<>(){
           @Override
           protected Integer initialValue() {
               return 0;
           }
       }
       
       // java8简写
       ThreadLocal<Ingeter> tl = ThreadLocal.withInitial(() -> 0);
       ```

   - 场景2 多数据源动态切换 （Mybatis / 多库项目）

     - 业务背景：

       - 后台系统存在多数据库：master主库（用户、部门基础数据）、slave从库（报表、查询只读数据）、business业务库（订单、设备业务数据）

     - 需求：同一个接口内，不同方法自动切换数据源，不用手动写连接、不用大量if判断

     - 核心方案：

       - 自定义注解@DataSource("库名")标记方法使用哪个库
       - ThreadLocal存储当前线程要使用的数据源标识
       - AOP切面拦截注解，往ThreadLocal存入数据源key
       - Mybatis动态数据源DynamicDatSource读取ThreadLocal的值切换库

     - 完整执行流程

       - 前端调用"/order/list"，执行getOrderList()方法
       - AOP切面匹配@DataSource("business")，执行before()
       - DataSourceContextHolder.setDataSource("business")，将字符串business存入当前请求线程的ThreadLocal
       - Mapper执行SQL前，`DynamicDataSource.determineCurrentLookupKey()` 被框架自动调用
       - 方法从 ThreadLocal 取出`business`，Spring 匹配 yml 中 business 数据源，建立对应库连接执行 SQL
       - 方法执行完毕，AOP `@After` 执行 `clear()`，调用`remove()`清空 ThreadLocal

     - 实现

       - 数据源上下文工具类（ThreadLocal核心）

       - ```java
         public class DataSourceContextHolder {
             // ThreadLocal:存储当前线程使用的数据源名称
             private static final ThreadLocal<String> DATA_SOURCE_TL = new ThreadLocal<>();
             
             // 设置当前线程数据源
             public static void setDataSource(String dbKey) {
                 DATA_SOURCE_TL.set(dbKey);
             }
             
             // 获取当前线程数据源
             public static String getDataSource() {
                 return DATA_SOURCE_TL.get();
             }
             
             // 清除数据源，防止线程池复用串库、内存泄漏
             public static void clear() {
                 DATA_SOURCE_TL.remove();
             }
         }
         ```

       - AOP切面：拦截注解，存入ThreadLocal

       - ```java
         import org.aspectj.lang.JionPoint;
         import org.aspectj.lang.annotation.After;
         import org.aspectj.lang.annotation.Aspect;
         import org.aspectj.lang.annotation.Before;
         import org.aspectj.lang.annotation.Pointcut;
         import org.aspectj.lang.reflect.MethodSignature;
         import org.springframework.steretype.Component;
         import java.lang.reflect.Method;
         
         @Aspect
         @Component
         public class DataSourceAspect {
             // 切点：所有加@DataSource注解的方法
             @Pointcut("@annotation(com.xxx.annotation.DataSource)")
             public void dataSourcePointCut() {}
             
             // 方法执行前，读取注解配置，存入ThreadLocal
             @Before("dataSourcePointCut()")
             public void before(JionPoint point) {
                 MethodSignature signature = (MethodSignature) point.getSignature();
                 Method method = signature.getMethod();
                 // 获取方法上的注解
                 DataSource dataSource = method.getAnnotation(DataSource.class);
                 // 将数据源标识存入当前线程ThreadLocal
                 DataSourceContextHolder.setDataSource(dataSource.value());
             }
             
             // 方法执行完毕，清除ThreadLocal，避免线程池污染
             @After("dataSourcePointCut()")
             public void after() {
                 DataSourceContextHolder.clear();
             }
             
         }
         ```

4. ThreadLocal 优势

   - **无锁、无并发竞争**：数据线程私有，不需要 synchronized/Lock
   - **避免参数透传**：不用在 Controller → Service → Mapper 层层传递参数
   - 数据隔离：异步子线程默认不继承（InheritableThreadLocal 可解决）

5. 高频坑点（面试必考）

   - 线程池复用线程，不 remove 导致内存泄漏 / 数据污染

     - 线程池线程不会销毁，会循环复用:
       - 第一次请求存入用户 A 数据，**未执行 remove ()**
       - 线程归还线程池，下次分配给新请求
       - 新线程 get () 读到上一次残留的用户 A 数据，造成数据错乱
       - value 强引用长期存在，GC 无法回收，引发内存泄漏
     - **解决方案**：请求 / 任务执行完毕，finally 中调用 `remove()`

   - 普通 ThreadLocal 子线程无法获取父线程数据

     - 主线程存入用户信息，新开子线程 `new Thread()`，子线程 ThreadLocal 为空，读取不到父线程数据

     - **解决方案**：使用 `InheritableThreadLocal`，创建子线程时自动拷贝父线程数据到子线程的 ThreadLocalMap

       - ```java
         ThreadLocal<LoginUser> tl = new InheritableThreadLocal<>();
         ```

   - key 弱引用不代表不会内存泄漏

     - 只有 key（ThreadLocal 引用）会被 GC 回收，但 Entry 里的 value 是强引用
     - 线程长期存活时，value 一直无法释放，内存持续占用
