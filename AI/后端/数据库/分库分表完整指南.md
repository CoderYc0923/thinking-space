## 一、概念与选型

### 1.1 什么是分库分表

分库分表是将单一数据库/数据表按照一定规则拆分为多个数据库/数据表的技术方案，目的是解决**单库单表数据量过大**导致的性能瓶颈。

### 1.2 触发时机

| 指标 | 阈值 |
|------|------|
| 单表行数 | > 500万（保守）~ 2000万行（激进） |
| 单库磁盘占用 | > 几百GB |
| 单库 QPS | > 2000 |
| 单库连接数 | > 500 |

### 1.3 垂直拆分 vs 水平拆分

| 维度 | 垂直拆分 | 水平拆分 |
|------|----------|----------|
| **拆分对象** | 按业务模块拆库 / 按字段拆表 | 按数据行拆表 |
| **解决什么问题** | 业务耦合严重，单库连接数打满 | 单表数据量过大 |
| **触发时机** | 微服务拆分阶段 | 单表千万级 |
| **核心挑战** | 跨库 JOIN、分布式事务 | 分片键选择、跨分片查询、数据迁移 |
| **常见场景** | 订单库、用户库、商品库各自独立 | 订单表按 user_id 哈希拆成 16 张表 |
| **复杂度** | 中等 | **高** |

> **金科玉律：先垂直，后水平。** 不要先把不同业务域的表混在一起，然后再做水平拆分——那是给自己挖坑。

#### 垂直拆分示例

```sql
-- 原始：所有表都在同一个库
mydb:
  ├── users
  ├── orders
  ├── products
  └── payments

-- 垂直拆库后
user_db:          order_db:         product_db:
  ├── users         ├── orders         ├── products
  ├── profiles      ├── order_items    ├── categories
  └── addresses     └── payments       └── inventory
```

#### 水平拆分示例

```sql
-- 原始：一张大表
orders (id, user_id, amount, status, created_at)  -- 2000万行

-- 水平拆分为 4 张表（按 user_id % 4）
orders_0  -- user_id % 4 == 0
orders_1  -- user_id % 4 == 1
orders_2  -- user_id % 4 == 2
orders_3  -- user_id % 4 == 3

-- 进一步分库（每个库 2 张表）
ds_0:
  ├── orders_0
  └── orders_1
ds_1:
  ├── orders_2
  └── orders_3
```

## 二、分片键设计

分片键是水平分表中**最重要的设计决策**，一旦确定，后期修改成本极高。

### 2.1 选型三原则

```
✅ 查询必须带分片键    → 否则需要全分片扫描（广播查询），性能灾难
✅ 数据分布均匀        → 避免热点分片扛不住
✅ 尽量是业务主键/外键  → 让 JOIN 在分片内完成，避免跨分片 JOIN
```

### 2.2 实战分析：订单表分片键选择

```sql
-- 订单表常见查询模式
SELECT * FROM orders WHERE user_id = ?;           -- 用户查自己订单（高频）
SELECT * FROM orders WHERE id = ?;                -- 查单个订单（高频）
SELECT * FROM orders WHERE status = ?;            -- 运营后台筛选（不带 user_id！）
SELECT * FROM orders WHERE merchant_id = ?;       -- 商家查订单（不带 user_id！）
```

| 分片键方案 | 精准命中 | 全分片扫描 | 评价 |
|-----------|---------|-----------|------|
| `user_id` | 用户查询、id 查询（需关联 user_id） | 商家查询、状态筛选 | C 端优先时最常用 |
| `merchant_id` | 商家查询 | 用户查询、状态筛选 | B 端优先时使用 |
| `id`（雪花） | 单行查询 | 所有列表查询 | 无业务偏向，列表查询全扫描 |

**常见折中方案：**

- 用 `user_id` 分片（覆盖 C 端高频查询）
- 对 `merchant_id` 查询和 `status` 筛选，额外建立 Elasticsearch 索引表
- 运营后台查询走 ES，不走数据库

### 2.3 分片键设计要点

```
✅ 避免用时间戳作为分片键 → 最新数据集中在一个分片，形成写入热点
✅ 避免用自增 ID 作为分片键 → 取模后数据分布不均
✅ 推荐：有业务含义且分布均匀的字段，或雪花算法生成的全局 ID
```

### 2.4 雪花算法（Snowflake）生成分片友好 ID

```
雪花 ID 结构（64位）：
┌─┬───────────────────────┬─────────────┬──────────┐
│1│    41位毫秒时间戳      │ 10位机器ID  │ 12位序列  │
└─┴───────────────────────┴─────────────┴──────────┘
 ↑    ↑                      ↑             ↑
保留 趋势递增（毫秒级）      分布式机器位   同毫秒内序列
```

**优势：** 全局唯一、趋势递增（对索引友好）、不依赖数据库、天然包含时间信息。

**Java 代码：**

```java
// Hutool 工具类
import cn.hutool.core.util.IdUtil;
long id = IdUtil.getSnowflake(1, 1).nextId();
```

**Node.js 代码：**

```javascript
// snowflake-id 库
const Snowflake = require('snowflake-id');
const snowflake = new Snowflake({ mid: 1, offset: 0 });
const id = snowflake.generate();  // 返回 string 类型
```

## 三、分片算法

| 算法 | 原理 | 数据均匀性 | 扩容难度 | 适用场景 |
|------|------|:---:|:---:|----------|
| **取模 MOD** | `分片 = id % N` | ⭐⭐⭐ | 🔴 全量迁移 | 数据量固定，短期不扩容 |
| **哈希取模** | `分片 = hash(key) % N` | ⭐⭐⭐ | 🔴 全量迁移 | key 非数值型，需要分布均匀 |
| **一致性哈希** | 哈希环 + 虚拟节点 | ⭐⭐ | 🟢 只迁移部分 | 分片数频繁变化 |
| **范围分片** | `user_id 1~1000万→分片0` | ⭐ | 🟢 加新分片即可 | 历史归档场景 |
| **日期分片** | 按月/年分片 | ⭐ | 🟢 加新分片即可 | 日志、流水等时序数据 |

### 3.1 最佳实践：预分片

不要等数据量大了再拆分，而是**一开始就创建足够多的分片**（如 16 或 32 个），扩容时只需迁移部分分片到新机器。

```
初始部署：
  机器A 承载分片 0-15（16个分片）

扩容时：
  机器A 承载分片 0-7
  机器B 承载分片 8-15
  → 只需迁移分片 8-15 的数据，无需全量重哈希
```

### 3.2 算法选择速查

| 需求 | 推荐算法 |
|------|----------|
| 数据均匀分布，短期不扩容 | MOD 取模 |
| 需要弹性扩容 | 一致性哈希 + 预分片 |
| 时序数据，按月归档 | 日期分片 |
| key 非数值（如字符串） | 哈希取模 |

## 四、中间件选型

### 4.1 主流方案对比

| 方案 | 类型 | Java/Node.js 通用 | 运维复杂度 | 适用规模 | 推荐指数 |
|------|------|:---:|:---:|------|:---:|
| **ShardingSphere-Proxy** | 独立代理 | ✅ | 中 | 中大型 | ⭐⭐⭐⭐⭐ |
| ShardingSphere-JDBC | 嵌入式 SDK | ❌ 仅 Java | 低 | 中小型 | ⭐⭐⭐ |
| MyCat | 独立代理 | ✅ | 高 | 大型 | ⭐⭐ |
| DBLE | 独立代理 | ✅ | 高 | 大型 | ⭐⭐ |
| Vitess | 独立代理 | ✅ | 很高 | 超大规模 | ⭐ |

### 4.2 推荐：ShardingSphere-Proxy

**为什么选它：**

1. **语言无关：** 对 Java 和 Node.js 应用无差别支持，只需连接代理地址即可
2. **Docker 友好：** 一个容器即可部署，配合 docker-compose 使用
3. **配置集中：** 分片规则在 Proxy 端配置，应用代码零改动
4. **社区活跃：** Apache 顶级项目，文档和生态完善

## 五、Docker Compose 部署 ShardingSphere-Proxy

### 5.1 docker-compose.yml

```yaml
version: '3.8'

services:
  # 模拟数据源 0
  mysql-ds0:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: root123
      MYSQL_DATABASE: orders_db
    ports:
      - "3307:3306"
    volumes:
      - ds0_data:/var/lib/mysql
    command: --default-authentication-plugin=mysql_native_password

  # 模拟数据源 1
  mysql-ds1:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: root123
      MYSQL_DATABASE: orders_db
    ports:
      - "3308:3306"
    volumes:
      - ds1_data:/var/lib/mysql
    command: --default-authentication-plugin=mysql_native_password

  # ShardingSphere-Proxy
  shardingsphere-proxy:
    image: apache/shardingsphere-proxy:5.5.1
    ports:
      - "3307:3307"
    volumes:
      - ./proxy-conf:/opt/shardingsphere-proxy/conf
    environment:
      - PORT=3307
    depends_on:
      - mysql-ds0
      - mysql-ds1

volumes:
  ds0_data:
  ds1_data:
```

### 5.2 proxy-conf/config-sharding.yaml

```yaml
# 注意：这是 ShardingSphere-Proxy 5.x 的配置格式
databaseName: orders_db

dataSources:
  ds_0:
    url: jdbc:mysql://mysql-ds0:3306/orders_db?useSSL=false&allowPublicKeyRetrieval=true
    username: root
    password: root123
    connectionTimeoutMilliseconds: 30000
    idleTimeoutMilliseconds: 60000
    maxLifetimeMilliseconds: 1800000
    maxPoolSize: 50
    minPoolSize: 1
  ds_1:
    url: jdbc:mysql://mysql-ds1:3306/orders_db?useSSL=false&allowPublicKeyRetrieval=true
    username: root
    password: root123
    connectionTimeoutMilliseconds: 30000
    idleTimeoutMilliseconds: 60000
    maxLifetimeMilliseconds: 1800000
    maxPoolSize: 50
    minPoolSize: 1

rules:
  - !SHARDING
    tables:
      orders:
        # 2 个数据源，每个源里 2 张表 → 共 4 个分片
        actualDataNodes: ds_${0..1}.orders_${0..1}
        
        # 分库策略：按 user_id % 2 决定去 ds_0 还是 ds_1
        databaseStrategy:
          standard:
            shardingColumn: user_id
            shardingAlgorithmName: database_inline
        
        # 分表策略：按 user_id % 2 决定去 orders_0 还是 orders_1
        tableStrategy:
          standard:
            shardingColumn: user_id
            shardingAlgorithmName: table_inline
        
        # 主键生成策略（可选）
        keyGenerateStrategy:
          column: id
          keyGeneratorName: snowflake
    
    shardingAlgorithms:
      database_inline:
        type: INLINE
        props:
          algorithm-expression: ds_${user_id % 2}
      table_inline:
        type: INLINE
        props:
          algorithm-expression: orders_${user_id % 2}
    
    keyGenerators:
      snowflake:
        type: SNOWFLAKE
```

## 六、数据迁移

从单表平滑迁移到分片表，需要经过四个阶段。

### 6.1 四阶段流程

```
┌─────────────────────────────────────────────────────────────┐
│  阶段一：存量迁移                                            │
│  原表 → 数据导出脚本 → 按分片键哈希写入各分片                 │
│  工具：DataX、Canal、或自己写脚本                            │
│  预计耗时：取决于数据量（千万级通常几小时）                   │
├─────────────────────────────────────────────────────────────┤
│  阶段二：增量双写（灰度过渡）                                │
│  应用层同时写老表和分片表 → 验证数据一致性                    │
│  持续时间：1~2 天确保无遗漏                                  │
│  关键：对比老表和分片表数据量、抽样校验内容                   │
├─────────────────────────────────────────────────────────────┤
│  阶段三：切流                                                │
│  读流量逐步切换到分片表 → 观察错误日志                       │
│  建议：先切 10% → 观察 → 50% → 观察 → 100%                  │
│  确认无误后，关闭老表写入                                    │
├─────────────────────────────────────────────────────────────┤
│  阶段四：清理                                                │
│  老表保留 1~2 周作为备份                                    │
│  确认无问题后归档或删除                                      │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 Node.js 迁移脚本示例

```javascript
const mysql = require('mysql2/promise');

async function migrate() {
  // 老库连接
  const oldPool = mysql.createPool({
    host: 'localhost', port: 3306, user: 'root',
    password: 'root123', database: 'orders_db',
  });

  // 分片数据源连接（直连，不走 Proxy）
  const dsPools = [
    mysql.createPool({ host: 'mysql-ds0', port: 3306, user: 'root', password: 'root123', database: 'orders_db' }),
    mysql.createPool({ host: 'mysql-ds1', port: 3306, user: 'root', password: 'root123', database: 'orders_db' }),
  ];

  const BATCH_SIZE = 1000;
  let lastId = 0;
  let totalMigrated = 0;

  console.log('开始存量数据迁移...');

  while (true) {
    // 从老表分批读取
    const [rows] = await oldPool.query(
      'SELECT * FROM orders WHERE id > ? ORDER BY id LIMIT ?',
      [lastId, BATCH_SIZE]
    );

    if (rows.length === 0) break;

    // 逐行写入分片
    for (const row of rows) {
      const dbIndex = row.user_id % 2;   // 分库规则
      const tableIndex = row.user_id % 2; // 分表规则

      await dsPools[dbIndex].query(
        `INSERT INTO orders_${tableIndex} (id, user_id, amount, status, created_at) 
         VALUES (?, ?, ?, ?, ?)`,
        [row.id, row.user_id, row.amount, row.status, row.created_at]
      );
    }

    lastId = rows[rows.length - 1].id;
    totalMigrated += rows.length;
    console.log(`已迁移 ${totalMigrated} 条，最后 ID: ${lastId}`);
  }

  // 关闭连接
  await oldPool.end();
  for (const pool of dsPools) await pool.end();

  console.log(`迁移完成，共 ${totalMigrated} 条数据`);
}

migrate().catch(console.error);
```

### 6.3 数据校验

```sql

-- 分片表（通过 Proxy 查询或各分片求和）
```

## 七、常见问题与解决方案

### 7.1 跨分片 JOIN

**问题：** 两张表分片键不同，JOIN 条件需要跨分片。

**解决方案：**

| 方案 | 描述 | 适用场景 |
|------|------|----------|
| 应用层组装 | 分别查两张表，在代码中做关联 | 数据量不大的情况 |
| 字段冗余 | 把关联字段冗余到同一张表 | 查询频率高，冗余字段变动小 |
| 广播表 | 小表在每个分片存一份全量 | 配置表、字典表等小表 |
| ES 宽表 | 在 Elasticsearch 中建宽表 | 复杂搜索场景 |

### 7.2 跨分片分页

**问题：** `LIMIT offset, size` 需要对所有分片的结果集做全局排序再分页。

**解决方案：**

```javascript
// 方案 A：内存合并（适合浅分页）
// 从每个分片取 offset + size 条 → 内存合并排序 → 取 offset 到 offset+size 区间
async function pagedQuery(page, pageSize) {
  const offset = (page - 1) * pageSize;
  // 从每个分片查询
  const promises = [0, 1, 2, 3].map(i => 
    query(`SELECT * FROM orders_${i} ORDER BY id LIMIT ${offset + pageSize}`)
  );
  const allResults = (await Promise.all(promises)).flat();
  // 内存排序后截取
  return allResults.sort((a, b) => b.id - a.id).slice(offset, offset + pageSize);
}
// 缺点：深分页时需要从每个分片取出大量数据，内存和网络开销大

// 方案 B：二次查询法（适合游标分页）
// 业务层维护分片的数据分布信息，或使用游标分页

// 方案 C：ES 承载分页
// 将订单数据同步到 ES，分页查询走 ES
```

### 7.3 分布式事务

**问题：** 跨分片的操作需要保证事务一致性。

**解决方案：**

| 方案 | 描述 | 复杂度 |
|------|------|:---:|
| 避免跨分片事务 | 设计层面让事务在单个分片内完成 | ⭐ |
| Seata AT 模式 | Java 生态的分布式事务中间件 | ⭐⭐⭐ |
| 最终一致性 | 本地事务 + 消息队列补偿 | ⭐⭐ |
| TCC 模式 | Try-Confirm-Cancel，需业务实现 | ⭐⭐⭐⭐ |

```
最佳实践：尽量避免跨分片事务。

怎么做？
  - 分片键统一：关联表使用相同的分片键，确保同一次业务操作的所有数据落在同一分片
  - 例如：订单表 user_id 分片，订单明细表也用 user_id 分片
    → 同一用户的订单和明细在同一个分片 → 单分片内事务
```

### 7.4 全局唯一 ID

| 方案 | 优点 | 缺点 |
|------|------|------|
| **雪花算法** ⭐推荐 | 全局唯一、趋势递增、高性能、不依赖 DB | 有时钟回拨风险 |
| Redis 自增 | 简单、连续递增 | 依赖 Redis 高可用 |
| 数据库号段模式 | 不依赖额外中间件 | 号段用完需更新，有性能瓶颈 |
| UUID | 完全无依赖 | 非递增、索引性能差、占用空间大 |

### 7.5 扩容

**最佳实践：预分片。**

```
不要等到 16 个分片不够用了才扩容到 32 个。
一开始就创建 32 或 64 个分片，部署在少量机器上。

扩容时：
  1. 新增物理机器
  2. 迁移部分分片到新机器
  3. 更新路由规则
  → 无需修改分片算法，无需全量数据重分布
```

### 7.6 广播表

```yaml
# ShardingSphere 配置广播表
rules:
  - !SHARDING
    broadcastTables:
      - t_config       # 配置表
      - t_district      # 地区字典表
    tables:
      orders:
        actualDataNodes: ds_${0..1}.orders_${0..1}
        # ...
```

广播表的特点：
- 每个数据源中都有全量数据
- 写操作同步到所有数据源
- 读操作从本地数据源获取，无需跨库
- 适用于：配置表、字典表等**数据量小、变更少**的表

## 附录：常见架构演进路径

```
单库单表
    │ 数据量增大
    ▼
读写分离（主从复制）
    │ 主库写入压力增大
    ▼
垂直分库（业务拆分）
    │ 单业务库表数据量过大
    ▼
水平分表（单库内多表）
    │ 单库磁盘/连接数瓶颈
    ▼
水平分库分表（多库多表）
    │ 持续增长
    ▼
分布式数据库（TiDB/OceanBase）
```

> **核心原则：** 不要在不需要的时候引入复杂度。每一种架构升级都伴随着巨大的运维成本。先用尽单库单表的优化手段，再考虑分库分表。
