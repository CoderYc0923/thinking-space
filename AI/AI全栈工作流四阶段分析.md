# AI 全栈开发工作流四阶段深度分析

> 整理时间：2026年6月25日

---

## 一、Vibe Coding（灵感编码）

### 1. 原理

Vibe Coding 的核心是**以意图驱动、用自然语言进行高模糊度的快速原型探索**。

开发者不用写具体代码，而是用自然语言向 AI 描述"我想要一个什么样的东西"，AI 直接生成可运行的初版。关键特征是：
- **低精度、高速度**：不追求代码质量，追求"看到东西"
- **对话式迭代**：不满意就继续用自然语言改，像聊天一样
- **零配置启动**：AI 自动选型技术栈、搭建环境

### 2. 优点

| 优点 | 说明 |
|------|------|
| 极低门槛 | 产品经理、设计师都能直接上手 |
| 极快出 demo | 几分钟内从想法变成可交互原型 |
| 激发创意 | 快速试错，低成本探索多种方案 |
| 降低心理启动成本 | 不用纠结技术选型就能开始 |

### 3. 缺点

| 缺点 | 说明 |
|------|------|
| 代码质量不可控 | 安全漏洞、性能问题、架构混乱 |
| 不可维护 | 生成代码缺乏统一规范，后续改动困难 |
| AI 幻觉风险 | 可能引用不存在的 API 或库 |
| 无测试覆盖 | 功能对错全靠人工肉眼判断 |
| 无法规模化 | 项目大了之后完全失控 |

---

## 二、Spec Coding（规格编码）

### 1. 原理

Spec Coding 的核心是**先写规格文档，再让 AI 根据规格生成代码**。

流程通常是：
1. 编写详细的 Spec 文档（功能需求、API 契约、数据模型、验收标准）
2. 将 Spec 作为 Prompt 喂给 AI
3. AI 严格按照 Spec 生成代码，不自行发挥
4. 代码需通过 Spec 中定义的测试用例

这是从"模糊对话"走向"精确契约"的转变。

### 2. 优点

| 优点 | 说明 |
|------|------|
| 代码可控 | 输出严格遵循规格，减少 AI 自行脑补 |
| 可验证 | Spec 就是验收标准，生成代码可自动对照 |
| 利于协作 | Spec 文档是人类和 AI 的共同语言 |
| 可追溯 | 任何代码都有对应的规格条目 |
| 减少幻觉 | 明确的约束条件有效抑制 AI 胡编 |

### 3. 缺点

| 缺点 | 说明 |
|------|------|
| 写 Spec 本身成本高 | 好的规格文档比写代码还难 |
| 灵活性下降 | 过于死板的 Spec 限制了 AI 的创造性 |
| 需领域专家 | 对写 Spec 的人要求极高 |
| 可能过设计 | 容易写出过度复杂的规格，产出冗余代码 |

---

## 三、Harness（保护套/护栏）

### 1. 原理

Harness 是在 AI 生成代码的**周围建立自动化验证和约束层**，类似一个"安全笼"。

它不是你手写测试，而是：
- **自动生成测试**：AI 理解 Spec 后，自动生成单元测试、集成测试、E2E 测试
- **自动跑 CI/CD**：每次 AI 产出代码，自动触发流水线
- **静态分析 + 安全扫描**：自动检测代码异味、漏洞
- **自动修复循环**：测试失败后 AI 自动修改代码直到通过

### 2. 优点

| 优点 | 说明 |
|------|------|
| 质量兜底 | 不管 AI 生成什么，护栏都能拦住坏的 |
| 自动化闭环 | 生成→验证→修复，全自动 |
| 信任建立 | 有测试通过的代码才敢上线 |
| 可复用 | 同一套 Harness 适用于多个 AI 生成的模块 |

### 3. 缺点

| 缺点 | 说明 |
|------|------|
| 搭建成本高 | Harness 本身需要精心设计和维护 |
| 过度约束 | 太严格的检查可能拖慢开发速度 |
| 测试质量依赖 | AI 生成的测试也可能有盲区 |
| 计算开销 | 持续跑完整测试套件成本不低 |

---

## 四、Loop（闭环循环）

### 1. 原理

Loop 是**全流程自动化闭环**：让 AI 不仅写代码，还能从生产环境中学习，持续自我优化。

核心循环链路：
```
想法 → Spec → AI生成代码 → Harness验证 → 部署上线 → 监控反馈 → 自动生成新Spec → 循环
```

关键特征：
- **生产数据反哺开发**：线上 bug、用户行为数据自动转化为新的 Spec
- **AI 自主排障**：告警触发后，AI 自动分析日志、定位问题、提交修复 PR
- **持续演进**：系统像生物一样自我适应

### 2. 优点

| 优点 | 说明 |
|------|------|
| 全自动化 | 从需求到运维全部打通 |
| 自我进化 | 系统越跑越好，不需人工频繁介入 |
| 快速响应 | 线上问题秒级发现、分钟级修复 |
| 数据驱动 | 基于真实数据决策，而非猜测 |

### 3. 缺点

| 缺点 | 说明 |
|------|------|
| 复杂度极高 | 需要基础设施（可观测、回滚、灰度）完备 |
| 失控风险 | AI 自动修改生产代码，需严格权限和安全设计 |
| 伦理与合规 | 自动决策可能触碰监管红线 |
| 成本巨大 | 只有大型团队或特定场景才值得投入 |

## 五、完整落地项目案例：AI 驱动的博客平台

下面用一个完整的项目案例，演示四阶段如何落地。

**项目背景**：开发一个带 AI 写作助手功能的个人博客平台。

### 阶段一：Vibe Coding — 快速出原型

**操作**：
- 用自然语言对 AI 说："帮我做一个博客网站，有首页文章列表、文章详情页，还有 AI 帮我写文章的功能。"
- AI 自动选择 Next.js + Tailwind CSS，生成完整可运行项目。
- 5 分钟后，一个本地能跑的博客就出来了。

**产出物**：
- 一个可交互的原型（非生产级代码）
- 用户登录/文章 CRUD 的基本页面

**验收方式**：
- 产品经理点一点，确认功能方向没问题

---

### 阶段二：Spec Coding — 精确化规格

基于原型，团队协作编写 Spec 文档（部分示例）：

```markdown
## Spec: 文章发布功能

### API 契约
- POST /api/posts
  - 请求体：{ title: string(1-200), content: string(1-50000), tags: string[] }
  - 需要 Bearer Token 认证
  - 返回 201 + { id, title, createdAt }
  - 错误：401 未认证、400 格式错误

### 数据模型
- posts 表：id (uuid), title, content, author_id, created_at, updated_at, published (bool)

### 验收标准
- [ ] 未登录用户调用接口返回 401
- [ ] 标题为空返回 400
- [ ] 发布成功后能在列表中看到新文章
```

**操作**：
- 将完整的 Spec 文档作为 Prompt 喂给 AI
- AI 生成严格符合 Spec 的后端代码（含参数校验、错误处理）
- 前端组件也按 Spec 中的交互说明生成

**产出物**：
- 符合 Spec 的 API 代码
- 前端组件（含表单验证）
- 数据库 migration 文件

---

### 阶段三：Harness — 建立护栏

**操作**：
- 基于 Spec，AI 自动生成测试套件：
  - 单元测试：测试认证中间件、参数校验逻辑
  - 集成测试：模拟 API 调用，验证 201/400/401 各种情况
  - E2E 测试：Playwright 脚本模拟用户登录→发布文章→查看文章
- 配置 GitHub Actions：
  - 每次 AI 生成代码 push 后自动跑全部测试
  - ESLint + Prettier 检查代码风格
  - npm audit 检查依赖安全漏洞
- 设置覆盖率阈值：低于 80% 则 CI 不通过

**Harness 结构示例**：
```yaml
# .github/workflows/ai-code-check.yml
name: AI Code Quality Gate
on:
  pull_request:
    types: [opened, synchronize]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run lint
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test -- --coverage
  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm audit --audit-level=high
```

**产出物**：
- 自动生成的测试套件
- CI/CD 流水线配置
- 质量门禁规则

---

### 阶段四：Loop — 闭环演进

**操作**：
- 博客部署上线后，接入可观测系统（Sentry + Grafana + 自定义埋点）
- 设定自动化规则：

```
规则 1：若 5xx 错误率 > 1%
  → 自动触发 AI 分析错误日志
  → AI 定位根因（如某个 API 未做限流导致数据库连接耗尽）
  → AI 自动生成修复代码 + 新 Spec（"增加 API 限流中间件"）
  → PR 自动创建，跑 Harness 测试
  → 测试通过后自动合并并灰度发布

规则 2：若文章编辑器的"AI 写作"功能使用率 < 5%
  → 自动分析用户行为数据
  → AI 推断：按钮不够显眼，用户没发现
  → AI 生成新 Spec："将 AI 写作按钮移至编辑区中央，增加引导气泡"
  → 走同样的代码生成→验证→部署流程
```

**闭环流程全景图**：
```
生产数据反馈
    ↓
AI 分析 → 生成新 Spec
    ↓
AI 生成代码
    ↓
Harness 自动验证
    ↓
灰度发布 → 监控数据再次反馈
    ↓
（循环）
```

## 六、四阶段对比总结

| 维度 | Vibe Coding | Spec Coding | Harness | Loop |
|------|------------|-------------|---------|------|
| 核心目标 | 快速看到东西 | 精确描述需求 | 质量兜底 | 持续自动进化 |
| 输入 | 模糊自然语言 | 结构化 Spec 文档 | Spec + 测试规则 | 生产数据 + Spec |
| 输出 | 粗糙原型 | 符合规格的代码 | 通过验证的代码 | 持续优化的系统 |
| 适用阶段 | 0→1 探索 | 1→10 开发 | 开发→上线 | 上线→迭代 |
| 人类角色 | 灵感提供者 | 规格制定者 | 质量架构师 | 系统监督者 |
| AI 角色 | 创意执行 | 精确翻译 | 验证 + 修复 | 分析 + 决策 + 执行 |
| 成熟度 | ⭐⭐ 较成熟 | ⭐⭐⭐ 快速演进 | ⭐⭐ 工具链丰富中 | ⭐ 早期探索 |

---

> **总结**：这四个阶段并非互相替代，而是**层层递进**的关系。Vibe Coding 帮你找到方向，Spec Coding 帮你锁定目标，Harness 保证你不翻车，Loop 让系统自己越跑越快。在真实项目中，你可能会在不同模块、不同阶段混合使用这四种范式。
